**1. DictusApp.swift -- Add auto-return after cold start URL handling:**

```swift
case "dictate":
    let isFromKeyboard = URLComponents(url: url, resolvingAgainstBaseURL: false)?
        .queryItems?.first(where: { $0.name == "source" })?.value == "keyboard"

    if isFromKeyboard {
        AppGroup.defaults.set(true, forKey: SharedKeys.coldStartActive)
        AppGroup.defaults.synchronize()
        DictusLogger.app.info("Cold start dictation from keyboard")
    }

    coordinator.startDictation(fromURL: true)

    // Auto-return: attempt to send user back to source app

        attemptAutoReturn()
    }
```

Add a private method `attemptAutoReturn()`:
```swift
private func attemptAutoReturn() {
    // Read source app scheme saved by keyboard before it opened us
    guard let sourceScheme = AppGroup.defaults.string(forKey: SharedKeys.sourceAppScheme),
          sourceScheme != "unknown" else {
        DictusLogger.app.info("No source app detected, showing swipe-back overlay")
        return
    }

    guard let returnURL = URL(string: sourceScheme) else { return }

    // IMPORTANT: Only attempt if we can open the URL
    if UIApplication.shared.canOpenURL(returnURL) {
        DictusLogger.app.info("Auto-returning to source app: \(sourceScheme)")
        UIApplication.shared.open(returnURL)
    } else {
        DictusLogger.app.info("Cannot open source app URL: \(sourceScheme)")
    }
}
```

WHY auto-return AFTER startDictation: Per research pitfall #3, audio session must be activated before switching apps. startDictation activates the session synchronously. Auto-return happens after.

WHY auto-return attempted FIRST before overlay shows: Per CONTEXT.md locked decision. The auto-return call is immediate. If it succeeds, the user barely sees the app. If it fails (unknown app or scheme not working), the swipe-back overlay (already rendered by MainTabView) is visible.

Add a private method `detectAndSaveSourceApp()`:
```swift
private func detectAndSaveSourceApp() {
    let defaults = AppGroup.defaults
    // Check known app schemes to guess which app the keyboard is open in
    // NOTE: canOpenURL from keyboard extension requires RequestsOpenAccess = true (Full Access)
    // and LSApplicationQueriesSchemes in DictusKeyboard Info.plist
    // LIMITATION: canOpenURL tells us if an app is INSTALLED, not which app is currently active.
    // We cannot reliably detect the source app from keyboard extension.
    // Strategy: write "unknown" -- the app will try all installed known schemes or show swipe overlay.
    defaults.set("unknown", forKey: SharedKeys.sourceAppScheme)
    defaults.synchronize()
}
```

WHY "unknown" default: Per research open question #1, there is no public API to detect which app the keyboard is currently serving. canOpenURL only tells if an app is installed, not which is active. Competitors face the same limitation. The swipe-back overlay is the reliable fallback.

FUTURE IMPROVEMENT: Could try checking all known schemes and if only ONE is installed, assume that's the source. But this is unreliable and not worth the complexity for Plan 03.
