# Phase 24.1: AOSP Trie Spell Correction Engine - Research

**Researched:** 2026-04-03
**Domain:** C++ compressed trie with Swift/ObjC++ interop for spell correction
**Confidence:** MEDIUM

## Summary

This phase replaces SymSpell (pure Swift, ~15 MiB for 10K words, edit distance 1) with an AOSP LatinIME-inspired compressed patricia trie (C++ with ObjC++ bridge, ~3-5 MiB for 100K+ words, edit distance 2, mmap'd). The AOSP LatinIME native layer is Apache 2.0 licensed and provides a battle-tested architecture for mobile spell correction with keyboard proximity scoring. The key innovation over SymSpell is the trie traversal approach: instead of pre-generating all edit-distance deletes (which explodes memory), the trie walks candidate paths during lookup, pruning by accumulated cost. Combined with mmap, this gives 10x vocabulary at 1/3 the memory.

The AOSP codebase is large (~50+ C++ files in native/jni/src/) with heavy Android/JNI dependencies. A full port is not feasible. The approach is to build a **custom compressed patricia trie** inspired by the AOSP architecture but purpose-built for Dictus: a Python script generates the binary .dict file, and a lean C++ reader (~5-10 files) traverses it with weighted edit distance scoring. The ObjC++ bridge exposes a clean API to Swift.

**Primary recommendation:** Build a custom binary trie format (simpler than AOSP v2) with a Python dict-builder, a focused C++ reader with proximity+accent scoring, and an ObjC++ wrapper. Do not attempt to port AOSP code verbatim -- extract the algorithms and data structure design, reimplement cleanly.

### Pattern 1: Binary Trie Format (Custom, AOSP-Inspired)

**What:** A compact binary patricia trie format designed for direct mmap access. Each node stores flags, character(s), frequency (for terminal nodes), and child offset.

**When to use:** Generating and reading the .dict files.

**Format specification (recommended):**

```
File layout:
  [Header: 32 bytes]
    magic:     4 bytes  "DTRI" (Dictus Trie)
    version:   2 bytes  uint16 (1)
    flags:     2 bytes  uint16 (reserved)
    node_count: 4 bytes uint32
    word_count: 4 bytes uint32
    max_freq:  4 bytes  uint32
    reserved:  12 bytes (zero-padded)

  [Trie nodes: variable length, sequential]
    Each node:
      flags:    1 byte
        bit 7-6: children pointer size (00=none, 01=2bytes, 10=3bytes, 11=4bytes)
        bit 5:   has multiple characters (patricia compression)
        bit 4:   is terminal (word end)
        bit 3:   reserved
        bit 2-0: reserved

      char_count: 1 byte (only if bit 5 set, else implicit 1)
      characters: N * 2 bytes (UTF-16 code units, N = char_count or 1)
      frequency:  2 bytes uint16 (only if terminal, 0-65535, higher = more frequent)
      children_offset: 0/2/3/4 bytes (relative offset to first child node)
      sibling present: implicit -- nodes at same level are sequential, ended by
                       a "last sibling" flag or explicit count
```

**Why simpler than AOSP v2:**
- No code point table indirection (use UTF-16 directly -- sufficient for French/English)
- No bigram/shortcut sections (Phase 25 scope)
- No v2/v4 versioning complexity
- Fixed 2-byte frequency (AOSP uses 1 byte 0-255; we want finer granularity)
- Children offset is absolute from file start (simpler than AOSP's relative offsets)

### Pattern 2: Weighted Edit Distance Traversal

**What:** Instead of SymSpell's pre-compute-all-deletes approach, walk the trie and accumulate edit costs. At each trie node, try: match (cost 0), substitution (cost varies by proximity), insertion (cost 1.0), deletion (cost 1.0), transposition (cost 1.0). Prune branches where accumulated cost exceeds max edit distance (2.0).

**When to use:** Core of `spellCheck()` implementation.

**Key insight from AOSP:** The `Suggest` class coordinates three strategies:
1. **Traversal** -- walks the trie depth-first
2. **Scoring** -- combines frequency + spatial distance + edit distance
3. **Weighting** -- applies per-error-type costs

**Scoring formula (recommended, AOSP-inspired):**
```
score = word_frequency_normalized * (1.0 - edit_cost / max_edit_distance)
```
Where `edit_cost` accounts for:
- Substitution: `proximity_cost(typed_key, candidate_key)` (0.0 to 1.0)
- Adjacent key substitution: ~0.4 (much cheaper than distant key)
- Accent substitution (e -> e with accent): ~0.1 (very cheap)
- Insertion: 1.0
- Deletion: 1.0
- Transposition: 0.7 (common typo, cheaper than substitution)

### Pattern 4: Proximity Map Construction

**What:** A lookup table mapping each key pair to a proximity cost (0.0 = same key, ~0.4 = adjacent, 1.0 = distant).

**How to build from existing layouts:**
The `FrenchKeyboardLayouts.swift` file defines key rows with exact key positions. Extract grid coordinates:

```
AZERTY Row 1: a(0,0) z(1,0) e(2,0) r(3,0) t(4,0) y(5,0) u(6,0) i(7,0) o(8,0) p(9,0)
AZERTY Row 2: q(0,1) s(1,1) d(2,1) f(3,1) g(4,1) h(5,1) j(6,1) k(7,1) l(8,1) m(9,1)
AZERTY Row 3:        w(1,2) x(2,2) c(3,2) v(4,2) b(5,2) n(6,2)
```

Row 2 is offset ~0.25 units right, Row 3 is offset ~0.75 units right (matching iOS stagger).

Proximity cost = `min(1.0, euclidean_distance(key_a, key_b) / threshold)` where threshold ~= 2.5 key widths.

For the C++ implementation, store as a flat 26x26 (or 36x36 with digits) float array, precomputed at compile time or loaded from proximity map data.

### Anti-Patterns to Avoid
- **Porting AOSP JNI layer:** The JNI bindings (com_android_inputmethod_latin_*.cpp) are Android-specific. Strip entirely.
- **Using AOSP v2/v4 format directly:** The format includes bigrams, shortcuts, code point tables, and versioning we don't need. Build simpler.
- **Allocating per-query:** The trie traversal should reuse a pre-allocated candidate buffer. No malloc in the hot path.
- **String copying in the bridge:** Pass UTF-8 C strings through the bridge, convert to NSString/Swift String only at the boundary.
- **Blocking main thread on load:** mmap itself is instant, but the first traversal will page-fault. Still do initial load on background queue.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Binary format read/write | Custom serialization logic | Packed structs + mmap | mmap gives zero-copy access; packed structs map directly to memory layout |
| Edit distance algorithm | Basic Levenshtein | Trie-integrated weighted DL distance | Standard Levenshtein doesn't support proximity weighting or trie pruning |
| Unicode normalization | Manual accent stripping | ICU / Foundation `precomposedStringWithCanonicalMapping` | Edge cases with combining characters, NFC vs NFD |
| Memory mapping | Manual file I/O + malloc | POSIX `mmap()` / `munmap()` | OS handles paging, zero load time, automatic memory pressure response |
| Keyboard coordinate extraction | Hardcoded values | Derive from FrenchKeyboardLayouts.swift definitions | Stays in sync with actual keyboard layout |

**Key insight:** The value of this phase is in the trie format + traversal algorithm + proximity scoring. Everything else (mmap, Unicode, file I/O) should use system primitives.

## Common Pitfalls

### Pitfall 1: Endianness and Struct Packing
**What goes wrong:** Binary .dict file generated on macOS (little-endian) reads incorrectly due to compiler padding or different byte order assumptions.
**Why it happens:** C++ structs may include padding bytes for alignment. Different compilers pack differently.
**How to avoid:** Use `__attribute__((packed))` or `#pragma pack(1)` for on-disk format structs. Explicitly use little-endian encoding (ARM64 iOS is little-endian, matching macOS).
**Warning signs:** Garbage data when reading first few nodes, frequency values that look wrong.

### Pitfall 2: mmap and App Extension Sandbox
**What goes wrong:** mmap fails on the .dict file in the keyboard extension bundle.
**Why it happens:** Keyboard extensions have restricted file access. The file must be in the extension's own bundle, not the main app bundle.
**How to avoid:** Add .dict files to the DictusKeyboard target (not DictusApp). Verify with `Bundle.main.path(forResource:ofType:)` from the keyboard extension context. Use `MAP_PRIVATE` flag.
**Warning signs:** mmap returns `MAP_FAILED`, errno = EACCES or ENOENT.

### Pitfall 3: Bridging Header in Extension Target
**What goes wrong:** Xcode can't find the bridging header, or the C++ files don't compile.
**Why it happens:** Extension targets have their own bridging header setting, separate from the app target. Also need to set "C++ Language Dialect" and "C++ Standard Library" build settings.
**How to avoid:** 
1. Create `DictusKeyboard-Bridging-Header.h` in the keyboard directory
2. Set `SWIFT_OBJC_BRIDGING_HEADER = DictusKeyboard/DictusKeyboard-Bridging-Header.h` in DictusKeyboard target build settings
3. Set `CLANG_CXX_LANGUAGE_STANDARD = c++17`
4. Set `CLANG_CXX_LIBRARY = libc++`
5. Add all .cpp and .mm files to DictusKeyboard target membership
6. Set Header Search Paths to include `$(SRCROOT)/DictusKeyboard/Vendored/AOSPTrie/include`
**Warning signs:** "Use of undeclared identifier", "Unknown type name" in Swift files that import the bridge.

### Pitfall 4: UTF-8 vs UTF-16 Boundary
**What goes wrong:** French accented characters (e with accent, c with cedilla, o with circumflex) corrupt when crossing the Swift/C++ boundary.
**Why it happens:** Swift String is UTF-8 internally (since Swift 5), NSString is UTF-16, C++ std::string is bytes. Accented characters are multi-byte in UTF-8.
**How to avoid:** 
- In .dict file: store characters as UTF-16 code units (2 bytes each, sufficient for BMP characters which covers all French/English).
- In C++ trie reader: work with uint16_t code units internally.
- In ObjC++ bridge: convert NSString to uint16_t array via `[string characterAtIndex:]` or `getCharacters:range:`.
- In Swift: NSString conversion handles this automatically.
**Warning signs:** "cafe" (4 chars) becomes 5 or 6 bytes, accent characters show as replacement characters.

### Pitfall 5: Memory Pressure and mmap Eviction
**What goes wrong:** After mmap, the OS evicts pages under memory pressure, causing page faults during spell check that add latency spikes.
**Why it happens:** iOS aggressively reclaims memory from keyboard extensions (50 MB limit). mmap'd pages are first to go since they can be re-read from disk.
**How to avoid:** 
- Keep working set small (~500 KB for typical lookups on 100K words).
- Use `madvise(MADV_SEQUENTIAL)` during initial load, `madvise(MADV_RANDOM)` for lookups.
- The trie's patricia compression naturally keeps hot paths (frequent words) near the top of the file, so frequently accessed pages stay resident.
**Warning signs:** Intermittent latency spikes (>10ms) on spell check after memory warnings.

### Pitfall 6: Accent Cost Too Aggressive
**What goes wrong:** Setting accent substitution cost too low (e.g., 0.0) causes "ou" to always correct to "ou with accent" (where), "a" to "a with accent" (has), etc.
**Why it happens:** These pairs differ only by accent, and if accent cost is free, the accented version always wins when it has higher frequency.
**How to avoid:** Set accent cost to ~0.1-0.2 (cheap but not free). This means accent correction only fires when the accented word has significantly higher frequency AND no other interpretation is closer. Test with: "ca" -> "ca with cedilla" (should correct), "ou" -> should NOT correct (context-dependent, deferred to Phase 25 n-grams).
**Warning signs:** Every short unaccented word gets "corrected" to its accented variant.

## Code Examples

### Example 1: mmap Loading (C++)
```cpp
// Source: POSIX mmap standard + iOS compatibility
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <unistd.h>

namespace dictus {

class Trie {
    const uint8_t* data_ = nullptr;
    size_t size_ = 0;
    int fd_ = -1;

public:
    bool loadMmap(const char* path) {
        fd_ = open(path, O_RDONLY);
        if (fd_ < 0) return false;

        struct stat st;
        if (fstat(fd_, &st) < 0) { close(fd_); fd_ = -1; return false; }
        size_ = static_cast<size_t>(st.st_size);

        data_ = static_cast<const uint8_t*>(
            mmap(nullptr, size_, PROT_READ, MAP_PRIVATE, fd_, 0));
        if (data_ == MAP_FAILED) {
            data_ = nullptr; close(fd_); fd_ = -1; return false;
        }
        madvise(const_cast<void*>(static_cast<const void*>(data_)),
                size_, MADV_RANDOM);
        return validateHeader();
    }

    void unload() {
        if (data_) { munmap(const_cast<uint8_t*>(data_), size_); data_ = nullptr; }
        if (fd_ >= 0) { close(fd_); fd_ = -1; }
        size_ = 0;
    }

    ~Trie() { unload(); }
};

} // namespace dictus
```

### Example 2: Trie Node Reading (C++)
```cpp
// Inspired by AOSP patricia_trie_reading_utils.cpp
namespace dictus {

struct TrieNode {
    uint8_t flags;
    uint16_t characters[32]; // max patricia node label length
    uint8_t char_count;
    uint16_t frequency;      // 0 if not terminal
    uint32_t children_offset; // 0 if no children
    bool is_terminal;
    bool has_children;
};

TrieNode readNode(const uint8_t* data, uint32_t& pos) {
    TrieNode node = {};
    node.flags = data[pos++];

    // Children pointer size from bits 7-6
    uint8_t children_ptr_size = (node.flags >> 6) & 0x03;
    node.has_children = (children_ptr_size > 0);

    // Multi-char patricia node
    bool multi_char = (node.flags >> 5) & 0x01;
    node.is_terminal = (node.flags >> 4) & 0x01;

    // Read characters
    if (multi_char) {
        node.char_count = data[pos++];
    } else {
        node.char_count = 1;
    }
    for (uint8_t i = 0; i < node.char_count; i++) {
        node.characters[i] = *reinterpret_cast<const uint16_t*>(&data[pos]);
        pos += 2;
    }

    // Read frequency if terminal
    if (node.is_terminal) {
        node.frequency = *reinterpret_cast<const uint16_t*>(&data[pos]);
        pos += 2;
    }

    // Read children offset
    node.children_offset = 0;
    if (children_ptr_size == 1) {
        node.children_offset = *reinterpret_cast<const uint16_t*>(&data[pos]);
        pos += 2;
    } else if (children_ptr_size == 2) {
        node.children_offset = data[pos] | (data[pos+1] << 8) | (data[pos+2] << 16);
        pos += 3;
    } else if (children_ptr_size == 3) {
        node.children_offset = *reinterpret_cast<const uint32_t*>(&data[pos]);
        pos += 4;
    }

    return node;
}

} // namespace dictus
```

```cpp
// Inspired by AOSP proximity_info.cpp
namespace dictus {

class ProximityMap {
    // 26 lowercase letters
    float distances_[26][26] = {};

    static float keyX(char c, bool azerty) {
        // AZERTY row offsets: row0=0.0, row1=0.25, row2=0.75
        // Returns normalized x position (0.0 to 9.0 for 10-key rows)
        static const char* azerty_rows[] = {"azertyuiop", "qsdfghjklm", "wxcvbn"};
        static const float row_offsets[] = {0.0f, 0.25f, 0.75f};
        // ... lookup character position
    }

    static float keyY(char c, bool azerty) {
        // Returns row number (0, 1, 2)
    }

public:
    void buildAZERTY() {
        for (int i = 0; i < 26; i++) {
            for (int j = 0; j < 26; j++) {
                char a = 'a' + i, b = 'a' + j;
                float dx = keyX(a, true) - keyX(b, true);
                float dy = keyY(a, true) - keyY(b, true);
                float dist = sqrtf(dx*dx + dy*dy);
                distances_[i][j] = std::min(1.0f, dist / 2.5f);
            }
        }
    }

    float cost(uint16_t a, uint16_t b) const {
        if (a == b) return 0.0f;
        if (a >= 'a' && a <= 'z' && b >= 'a' && b <= 'z') {
            return distances_[a - 'a'][b - 'a'];
        }
        return 1.0f; // unknown key pair
    }
};

} // namespace dictus
```

### Example 4: Accent Cost Matrix
```cpp
// Accent substitution costs -- lower = more likely correction
float accentCost(uint16_t from, uint16_t to) {
    // Base letter to accented variant: very cheap
    // e(0x65) <-> e-acute(0xE9), e-grave(0xE8), e-circumflex(0xEA), e-diaeresis(0xEB)
    // a(0x61) <-> a-grave(0xE0), a-circumflex(0xE2)
    // u(0x75) <-> u-grave(0xF9), u-circumflex(0xFB)
    // o(0x6F) <-> o-circumflex(0xF4)
    // i(0x69) <-> i-circumflex(0xEE), i-diaeresis(0xEF)
    // c(0x63) <-> c-cedilla(0xE7)

    // Check if from/to pair is a base-letter <-> accented-variant
    // Cost: 0.15 (cheap enough to prefer accented, but not free)
    static const std::unordered_map<uint16_t, uint16_t> base_letter = {
        {0xE9, 'e'}, {0xE8, 'e'}, {0xEA, 'e'}, {0xEB, 'e'},
        {0xE0, 'a'}, {0xE2, 'a'},
        {0xF9, 'u'}, {0xFB, 'u'},
        {0xF4, 'o'},
        {0xEE, 'i'}, {0xEF, 'i'},
        {0xE7, 'c'},
    };

    auto it_from = base_letter.find(from);
    auto it_to = base_letter.find(to);

    // Unaccented -> accented (most common: user types "cafe", should get "cafe with accent")
    if (it_to != base_letter.end() && it_to->second == from) return 0.15f;
    // Accented -> unaccented (less common but valid)
    if (it_from != base_letter.end() && it_from->second == to) return 0.15f;
    // Accented -> different accented of same base (rare)
    if (it_from != base_letter.end() && it_to != base_letter.end()
        && it_from->second == it_to->second) return 0.2f;

    return -1.0f; // Not an accent pair -- use normal substitution cost
}
```

| SymSpell (pre-computed deletes) | Trie traversal with pruning | Always been the standard for mobile keyboards | 10x vocabulary at 1/3 memory; proximity scoring impossible with SymSpell |
| ObjC++ bridging (iOS) | Swift/C++ native interop (Xcode 15+) | WWDC 2023 | Native interop is newer but ObjC++ is proven and has no edge-case surprises in extensions |
| AOSP v2 Patricia trie | AOSP v4 format (on Android) | ~2014 | v4 adds dynamic updates, mutable dictionaries. We only need v2-style read-only. |
| Single-byte frequency (0-255) | 16-bit frequency | Our custom format | Finer granularity for ranking French words with similar frequencies |

**Deprecated/outdated:**
- SymSpell for mobile keyboards: Memory explosion at edit distance 2 makes it impractical for 100K+ word dictionaries on memory-constrained devices.
- AOSP v4 format for read-only use: Adds complexity (multiple files, dynamic update support) that we don't need.

## Open Questions

1. **Sibling encoding in binary format**
   - What we know: AOSP uses a "last sibling" flag bit to mark the end of a sibling list. An alternative is to store sibling count.
   - What's unclear: Which encoding is more compact for our 100K word dictionary.
   - Recommendation: Use "child count" byte at the start of each node array (simpler to implement, marginal size difference at 100K words).

2. **Exact file size for 100K French words**
   - What we know: AOSP dictionaries for ~100K words are typically 1-4 MiB. SymSpell was 15 MiB for 10K words. Patricia compression significantly reduces node count vs standard trie.
   - What's unclear: Exact size depends on patricia compression ratio for French vocabulary.
   - Recommendation: Build the dict-builder first, measure actual size, adjust if needed. Budget: 5 MiB max per language.

3. **Apostrophe handling in trie**
   - What we know: French contractions (l'homme, qu'il, aujourd'hui) split at apostrophe. Current SymSpellEngine checks only the part after the last apostrophe.
   - What's unclear: Should the trie contain full apostrophe words, or should splitting remain in the Swift wrapper?
   - Recommendation: Keep splitting in Swift wrapper (AOSPTrieEngine.swift), same as current SymSpellEngine pattern. Trie only stores base words.

4. **Xcode project file modifications**
   - What we know: Adding C++ files, bridging header, and build settings requires editing project.pbxproj.
   - What's unclear: Whether Xcode GUI or manual pbxproj editing is more reliable for extension targets.
   - Recommendation: Use Xcode GUI to add files and set build settings. Document the exact settings for reproducibility.
