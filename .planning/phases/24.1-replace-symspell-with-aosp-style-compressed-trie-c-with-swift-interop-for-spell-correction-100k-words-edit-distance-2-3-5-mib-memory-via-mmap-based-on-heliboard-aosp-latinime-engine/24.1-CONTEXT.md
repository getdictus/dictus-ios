# Phase 24.1: AOSP Trie Spell Correction Engine - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the SymSpell spell correction engine with an AOSP LatinIME-style compressed trie (C++ with Swift interop via ObjC++ bridging) for production-grade correction quality. Target: 100K+ words per language, edit distance 2, keyboard proximity scoring, accent-aware cost model, ~3-5 MiB on disk with mmap loading (~500 KB working set). SymSpell is removed entirely (clean replace, no fallback). UITextChecker completions, FrequencyDictionary, UserDictionary, suggestion bar layout, and autocorrect rejection behavior are all preserved unchanged.

Requirements: PRED-03 (supersedes Phase 24's SymSpell approach with higher quality engine)

</domain>

<decisions>
## Implementation Decisions

### C++ integration strategy
- **ObjC++ bridging header** approach (classic, battle-tested). No Swift/C++ native interop.
- Write thin ObjC++ wrapper (.mm files) that calls C++, expose ObjC API via bridging header to Swift.
- **Minimal port** of AOSP LatinIME C++ code: only trie traversal + weighted edit distance scorer. Strip Android-specific code (JNI, Java callbacks, Android logging). Target ~5-10 C++ source files.
- **File location:** `DictusKeyboard/Vendored/AOSPTrie/` — same vendoring pattern as current SymSpell.
- **Clean replace:** SymSpell code removed entirely when trie is integrated. No fallback/feature flag. If trie fails validation, git revert.

### Dictionary format & build pipeline
- **Binary .dict format:** AOSP-style compressed trie nodes, designed for direct mmap access.
- **Offline Python script** generates .dict binaries from source data. Run manually when dictionaries change, commit .dict files to repo. No build-time dependency.
- **Source data:** Reuse Phase 24's curated Lexique 3.83 (FR) and COCA/Google 1T (EN) frequency data, expanded from 10K to 100K+ words per language.
- **mmap loading at runtime:** Memory-map .dict file directly from app bundle. OS pages in only accessed parts — working set ~500 KB even for 100K+ words. Zero load time (no parsing step).
- **FrequencyDictionary stays on JSON** — only used for UITextChecker completion ranking, separate concern. Trie scope is spell correction only.

### Scoring & correction quality
- **Frequency + keyboard proximity scoring:** Combine word frequency with keyboard proximity cost (adjacent key substitution costs less than distant key substitution). Core AOSP innovation over SymSpell's fixed edit distance.
- **Proximity maps for AZERTY + QWERTY:** Both layouts supported. Active layout (from App Group settings) determines which proximity map is used. Switch map on layout change.
- **Edit distance 2:** Trie + mmap makes distance 2 affordable (unlike SymSpell which exploded in memory). Catches more typos including double-letter errors ("bonojur" -> "bonjour").
- **Accent-aware scoring:** Accent substitutions (e->e with accent, a->a with accent) have lower cost than regular character substitutions. "cafe" -> "cafe with accent" wins decisively over "cafe" -> "case". Makes corrections feel natural for French.
- **No bigram context yet** — that's Phase 25 n-grams. Scoring uses frequency + proximity + accent cost only.

### Migration scope
- **What stays unchanged:**
  - UITextChecker for completions (partial word suggestions)
  - FrequencyDictionary (JSON, ranks UITextChecker results)
  - UserDictionary (App Group persistence, repetition/rejection learning)
  - SuggestionBarView layout (verbatim | correction | alternative)
  - Autocorrect rejection behavior (undo + re-space doesn't re-apply)
  - SuggestionState async coalescing pattern
  - DictusKeyboardBridge handleSpace/handleBackspace autocorrect flow
- **What changes:**
  - SymSpellEngine replaced by new AOSPTrieEngine (ObjC++ wrapper around C++ trie)
  - TextPredictionEngine.spellCheck() delegates to AOSPTrieEngine instead of SymSpellEngine
  - Vendored SymSpell/ directory removed, replaced by Vendored/AOSPTrie/
  - Dictionary files: JSON frequency files replaced by binary .dict files for spell correction (JSON kept for FrequencyDictionary)
  - New: keyboard layout proximity maps (AZERTY + QWERTY)
  - New: bridging header + ObjC++ wrapper files added to Xcode project
- **UserDictionary integration:** Separate lookup layer. User words stay in Swift Set/Dictionary, checked before trie lookup. Avoids modifying the mmap'd trie at runtime. Two-pass lookup: user dict first, then trie.
- **French overrides ('ca' -> 'ca with cedilla') stay in Swift** wrapper layer. Trie is language-agnostic.
- **Swift API:** Richer than drop-in replacement. Expose: spellCheck(), wordExists(), getFrequency(), proximity score, multiple candidates with scores. Supports future Phase 25 n-gram integration.
- **Vocabulary:** 100K words per language (massive jump from current 10K).

<code_context>
## Existing Code Insights

### Reusable Assets
- `TextPredictionEngine`: Clean orchestrator with `spellCheck()` and `suggestions()` separation. Only `spellCheck()` changes.
- `SuggestionState`: Async coalescing with DispatchWorkItem, `rejectedWords: Set<String>`, `AutocorrectState`. Fully reusable.
- `UserDictionary`: App Group persistence with repetition/rejection learning. Becomes the "first pass" lookup before trie.
- `FrequencyDictionary`: JSON loader for `[String: Int]`. Stays as-is for UITextChecker ranking.
- `FrenchKeyboardLayouts.swift`: Contains key positions for AZERTY layout. Can extract x,y coordinates for proximity map generation.

### Established Patterns
- Vendored code in `DictusKeyboard/Vendored/` with dedicated subdirectory per library
- Background queue loading (`com.pivi.dictus.symspell-load`, `.userInitiated` QoS) — same pattern for trie init
- One language in memory at a time. `setLanguage()` drops and rebuilds.
- French overrides in Swift wrapper layer (language-specific logic stays in Swift)

### Integration Points
- `TextPredictionEngine.spellCheck()` — swap SymSpellEngine for AOSPTrieEngine
- `TextPredictionEngine.init()` and `setLanguage()` — init/reload trie instead of SymSpell
- `SymSpellEngine.injectUserWord()` — replaced by separate UserDictionary lookup pass
- Xcode project: add C++ source files, ObjC++ wrapper, bridging header to DictusKeyboard target
- Build settings: enable C++ compilation, set header search paths for AOSPTrie/

</code_context>

<specifics>
## Specific Ideas

- Pierre wants "aux petits oignons" — the best possible keyboard correction quality, as close to Apple native as feasible, fully built on open-source (HeliBoard/AOSP LatinIME engine).
- Standard mobile patterns — follow iOS/Android conventions, no exotic UX.
- The mmap approach is key: 100K+ words with ~500 KB working set vs SymSpell's 15 MiB for 10K words. This is why the migration is worth the C++ complexity.
- Accent-aware scoring is important for French feel: e->e-accent should be cheap, making "cafe" -> "cafe with accent" a clear winner.

</specifics>
