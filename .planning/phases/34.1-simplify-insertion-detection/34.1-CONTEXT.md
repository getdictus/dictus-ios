# Context — Phase 34.1 Simplify Insertion Detection

## Why this phase exists

Phase 34 (Plans 01/02/03) shipped a detect → retry → escalate pattern for the silent insertion bug #118. Real-device testing on 2026-04-16 revealed this introduces **more user-visible bugs than it fixes** because the base rate of true #118 failures is roughly 1-in-hundreds while the classifier false-positive rate — amplified by blind retries — is much higher.

User summary: *"ce bug d'insertion de texte, je ne l'ai eu qu'une seule fois sur des centaines et des centaines d'utilisations. Donc on a peut-être un peu over-ingénéré le problème."*

Goal of 34.1: simplify to telemetry-only. Keep the probe logs (they're genuinely useful for diagnosing future occurrences). Drop the retry loop + user-facing escalation (banner + haptic + LiveActivity `.failed`). Rely on the pre-existing HomeView transcription card as the recovery surface — augmented by Plan 34-02's App Group fallback when the keyboard classifies a true failure and preserves the key.

### Category A — empty-field false-positive proxyDead

Log at 08:15:38:
```

keyboardInsertRetry  path=warmDarwin attempt=0 reason=proxyDead
keyboardInsertProbe path=warmDarwin attempt=1 transcriptionCount=407 hasFullAccess=true hasTextBefore=true hasTextAfter=true beforeCount=407 afterCount=814 ...

```

Analysis:
- Attempt 0: `beforeCount=-1` (nil before-context on empty field) + `afterCount=407` exactly equals `transcriptionCount=407` + `hasText` flipped false→true.
- This is a clean empty-field success. The classifier returned `.proxyDead` because `InsertionClassifier.swift:54` checks `beforeCount < 0` before considering the `hasText` transition.
- Retry attempt 1 re-inserted: final field has 2× the text. Classifier saw delta=407, returned `.success`. But the user now has duplicated text.

### Category B — window-truncation false-positive deltaMismatch

Log at 08:15:54:
```

keyboardInsertRetry  path=warmDarwin attempt=0 reason=deltaMismatch

Analysis:
- Attempt 0: field had 816 chars of visible context before insertion. After inserting 315, `documentContextBeforeInput` returns only 500 chars — iOS truncated its returned window. Delta = 500 - 816 = **-316**.
- Classifier's default path returns `.deltaMismatch` (line 87). But hasTextAfter=true and insertion actually worked.
- Retry re-inserted: field now has 2× the text.

### Category C — terminal failure cascade after false-positives

Log at 08:16:34 (4 attempts all duplicating):
```
attempt=0 before=817 after=500 transcriptionCount=647 → deltaMismatch → retry
attempt=1 before=500 after=500                       → silentDrop     → retry
attempt=2 before=500 after=500                       → silentDrop     → retry
attempt=3 before=500 after=500                       → silentDrop

Analysis:
- Attempt 0 triggered Category B (window truncation, false-positive deltaMismatch).
- Retries re-inserted the same 647-char text 3 more times — total 4× in the field.
- Each retry couldn't grow the `documentContextBeforeInput` window (stuck at the 500-char iOS cap), so the classifier kept returning `.silentDrop`.
- Terminal failure fired red banner + insertionFailed haptic — while the user's field actually contained **4× their dictated text**.

## Design decisions

### D1: Fix classifier with success-first priority

New rule order (authoritative-success checks run before failure checks):

1. `hasTextBefore == false && hasTextAfter == true` → `.emptyFieldSuccess` — hasText transition is authoritative regardless of context-window state.
2. Both counts ≥ 0 and `delta == transcriptionUtf16Count` → `.success`.
3. Both counts ≥ 0 and `delta > 0 && delta < transcriptionUtf16Count` → `.windowedSuccess`.
4. Both counts ≥ 0 and `delta < 0 && hasTextAfter == true` → `.windowedSuccess` (iOS truncated its window, insertion succeeded).
5. `afterCount >= 0 && hasTextAfter == true && beforeCount < 0` → `.windowedSuccess` (first insertion into non-empty field where proxy gave us no pre-context).
6. `delta == 0 && beforeCount > 0 && beforeCount < 400 && hasTextBefore == hasTextAfter` → `.silentDrop` — narrow true-failure window (below truncation cap, no growth, no hasText transition). This is the ONLY retry/preserve candidate.
7. All other cases → `.windowedSuccess` (benefit of doubt — do not punish the user for iOS oddness).

### D2: Remove retry loop

`InsertTranscriptionHelper.insertTranscription` calls `proxy.insertText` exactly once. No `DispatchQueue.main.asyncAfter` retry scheduling. No `scheduleNextOrFail` function.

### D3: Remove user-facing escalation

- No red banner via `ToolbarView.statusMessage` from the insertion failure path
- No `HapticFeedback.insertionFailed()` call on failure
- No `LiveActivity` transition to `.failed` from the keyboard insertion flow

Keep these APIs available in DictusCore (shipped in 34-01) for potential future use, but don't call them from 34.1-modified paths.

### D4: Keep probe telemetry, privacy-audited

`keyboardInsertProbe` log event continues to emit on every insertion attempt. `keyboardInsertRetry` is dead code after D2 — leave the case definition in `LogEvent` for future re-enablement, but `InsertTranscriptionHelper` never emits it. `keyboardInsertFailed` still emits on the narrow true-failure case (D1 rule 6) — purely for telemetry, no UI side-effect.

### D5: Preserve App Group on true failure, clear on success

- On `.success | .windowedSuccess | .emptyFieldSuccess`: clear `SharedKeys.lastTranscription` (existing duplicate-prevention behavior).
- On `.silentDrop` (the only true failure after D1 simplification): preserve `SharedKeys.lastTranscription`. HomeView's `recoverableTranscription` (Plan 34-02) will surface the lost text.
- `.deltaMismatch` / `.proxyDead` cases are eliminated by D1's new rule ordering.

## Non-goals for 34.1

- Re-introducing retries with smarter heuristics. If production data later shows measurable silent drops, revisit with evidence.
- Re-introducing the banner. Same rationale — evidence-based, not speculative.
- Fixing the `.recording → .failed` LiveActivity edge gap logged in STATE.md Known Gaps. Not blocking.
- Changing Plan 34-02's `HomeView.recoverableTranscription` behavior. It already works; it just needs the keyboard to actually preserve the App Group key on failure (D5 delivers that).
- Closing Plan 34-04 (test matrix). Will resume after 34.1 ships, with a light 15-run sanity protocol instead of 50-run.
