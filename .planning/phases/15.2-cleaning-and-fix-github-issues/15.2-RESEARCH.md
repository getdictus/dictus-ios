## Architecture Patterns

### Current Architecture Overview

The dictation pipeline uses cross-process IPC between three targets:

```
DictusApp (main app)
  DictusApp.swift          -- scenePhase handler, URL routing
  DictationCoordinator     -- dictation lifecycle, audio engine, transcription
  LiveActivityManager      -- Dynamic Island / Live Activity state

DictusKeyboard (extension)
  KeyboardState            -- observes App Group + Darwin notifications
  KeyboardRootView         -- RecordingOverlay conditional rendering

DictusCore (shared framework)
  BrandWaveform            -- waveform visualization (TimelineView + Canvas)
  SharedKeys / AppGroup    -- cross-process UserDefaults
  DarwinNotificationCenter -- cross-process notifications

DictusWidgets (widget extension)
  DictusLiveActivity       -- Dynamic Island UI
  StopStandbyIntent        -- LiveActivityIntent for DI buttons
```

### Entry Paths Into Recording

Five distinct paths can start a recording:

| Entry Path | Mechanism | Code Location |
|------------|-----------|---------------|
| In-app mic button | Direct call | `RecordingView` -> `coordinator.startDictation()` |
| Keyboard mic (warm) | Darwin notification | `KeyboardState.startRecording()` -> Darwin -> `DictationCoordinator` |
| Keyboard mic (cold) | URL scheme `dictus://dictate?source=keyboard` | `DictusApp.handleIncomingURL()` |
| DI expanded mic | `Link(destination: "dictus://dictate")` | Widget -> URL -> `DictusApp.handleIncomingURL()` |
| Lockscreen mic | Same `Link(destination: "dictus://dictate")` | Widget -> URL -> `DictusApp.handleIncomingURL()` |

**Critical insight for #41:** DI tap and lockscreen paths use `dictus://dictate` WITHOUT `?source=keyboard`, so they bypass cold start detection. They go through `startDictation(fromURL: true)` but do NOT clear `lastResult` first.

### LiveActivityManager Current State Flow

```
startStandbyActivity() -> .standby
transitionToRecording() -> .recording
transitionToTranscribing() -> .transcribing
endWithResult() -> .ready -> (1s auto-dismiss) -> returnToStandby() -> .standby
endWithFailure() -> .failed -> (3s auto-dismiss) -> returnToStandby() -> .standby
stopStandbyActivity() -> end activity entirely
```

**No formal state machine** -- transitions are method calls with no guards against invalid transitions (e.g., recording -> recording, or standby -> transcribing).

## Bug Analysis

### Bug Cluster 1: #37 -- Cold Start Keyboard Freeze

**Status of proposed fixes:** The 3 fixes from #37 have been APPLIED to the codebase (confirmed by reading DictusApp.swift lines 65-87 and LiveActivityManager.cleanupStaleActivities lines 324-348 and DictationCoordinator.cleanupRecordingKeys lines 432-441).

- **Fix 1 (guard .background handler)**: Applied -- `isRecordingActive` guard at lines 65-87 of DictusApp.swift
- **Fix 2 (protect cleanupStaleActivities race)**: Applied -- `currentSessionActivityID` capture at line 329 of LiveActivityManager.swift
- **Fix 3 (clear coldStartActive in cleanupRecordingKeys)**: Applied -- lines 438-439 of DictationCoordinator.swift

**Remaining issue (from March 16 test comment):** Waveform animation is broken on BOTH warm and cold starts. App Group data flows correctly (energy count increases), but BrandWaveform visual animation does not render in the keyboard extension. The `TimelineView(.animation)` / `onChange(of: energyLevels)` fallback may be insufficient.

**Root cause hypothesis:** After keyboard extension process suspension/resumption, `TimelineView`'s internal `CADisplayLink` does not restart. The `onChange(of: energyLevels)` fallback calls `updateDisplayLevels()` but this only lerps one step per data change (~5Hz), which may produce imperceptibly small visual changes that SwiftUI optimizes away. The Canvas redraws depend on `displayLevels` @State changes, and SwiftUI may coalesce rapid small changes.

**Confidence:** MEDIUM -- the data flow is confirmed working (App Group writes arrive), so the problem is purely in the SwiftUI rendering pipeline within the keyboard extension.

### Bug #42: Dynamic Island Desync After Chaining

**Root cause:** No state machine guards in LiveActivityManager. After multiple recording cycles:
1. `autoDismissTask` (1s delay in `endWithResult`) can fire between the user's next `transitionToRecording()` call, flashing DI back to standby mid-recording
2. Concurrent `Activity.update()` calls can arrive out of order (Swift concurrency does not guarantee ordering of detached Tasks)
3. `startStandbyActivity()` can recover a "stale" activity that was actually transitioning

**State machine design recommendation:**

```swift
enum LiveActivityPhase: String {
    case idle       // No activity exists
    case standby    // Activity exists, waiting for user
    case recording  // Active recording
    case transcribing // Processing audio
    case ready      // Showing result
    case failed     // Showing error
}

// Valid transitions:
// idle -> standby (startStandbyActivity)
// standby -> recording (transitionToRecording)
// standby -> idle (stopStandbyActivity)
// recording -> transcribing (transitionToTranscribing)
// recording -> standby (cancelDictation)
// transcribing -> ready (endWithResult)
// transcribing -> failed (endWithFailure)
// ready -> standby (auto-dismiss or new recording)
// failed -> standby (auto-dismiss)
// ready -> recording (new recording started before auto-dismiss)
```

Guard every transition method with valid source states. Log and reject invalid transitions.

**Confidence:** HIGH -- the symptoms match classic state machine absence.

### Bug #41: Transcription Card Not Cleared

**Root cause:** When recording starts via DI tap or lockscreen notification (`dictus://dictate` without `?source=keyboard`):
1. `handleIncomingURL` calls `coordinator.startDictation(fromURL: true)`
2. `startDictation()` does NOT clear `lastResult`
3. `RecordingView` shows `coordinator.lastResult` at line 288 -- the old result persists

When recording starts via in-app mic button, `RecordingView` is already visible and its state naturally resets because the view is already mounted and `coordinator.status` transitions through `.idle` -> `.recording`.

**Fix approach:** Clear `lastResult` at the beginning of `startDictation()`, OR add a `resetForNewRecording()` method that clears `lastResult`, `bufferEnergy`, `bufferSeconds`, and call it from all entry paths.

**Confidence:** HIGH -- the entry path difference is clear from code reading.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State machine validation | Manual if/else guards in each method | Enum-based state machine with transition table | Exhaustive, testable, single source of truth |
| Activity lifecycle sync | Ad-hoc nil checks on currentActivity | System-level `Activity.activities` reconciliation at key points | iOS can kill activities independently |
| Waveform animation recovery | More CADisplayLink workarounds | `onChange(of: energyLevels)` with forced objectWillChange.send() | Keyboard extension process lifecycle is unreliable |

## Common Pitfalls

### Pitfall 1: ActivityKit Update Budget Exhaustion
**What goes wrong:** Sending too many `Activity.update()` calls causes iOS to silently drop updates, making DI appear frozen.
**Why it happens:** Each recording cycle sends waveform updates at 1Hz. After many cycles, iOS may throttle.
**How to avoid:** Maintain the 1Hz throttle in `updateWaveform()`. When transitioning phases, cancel pending waveform updates before sending phase transition update.
**Warning signs:** DI shows correct phase text but waveform bars freeze.

### Pitfall 2: Task Cancellation in Auto-Dismiss
**What goes wrong:** `autoDismissTask` from a previous `endWithResult()` fires during the next recording, resetting DI to standby.
**Why it happens:** `transitionToRecording()` cancels `autoDismissTask` but the cancel may not propagate if the Task is already past its `isCancelled` check.
**How to avoid:** Add a `guard !Task.isCancelled` check AFTER the sleep AND before the `returnToStandby()` call. Also add phase validation: only return to standby if current phase is still `.ready` or `.failed`.
**Warning signs:** DI briefly flashes "On" during an active recording.

### Pitfall 3: SwiftUI Rendering Coalescing in Extensions
**What goes wrong:** `@State` updates in rapid succession get coalesced by SwiftUI, so intermediate waveform frames are never rendered.
**Why it happens:** Keyboard extensions run with reduced priority. SwiftUI may batch multiple `displayLevels` updates into a single frame.
**How to avoid:** Ensure each `updateDisplayLevels()` produces a meaningfully different array (not micro-changes below SwiftUI's diff threshold). Consider using `objectWillChange.send()` explicitly if needed.
**Warning signs:** Waveform data arrives (logged) but bars don't move.

### Pitfall 4: scenePhase .background Firing During URL Scheme Transitions
**What goes wrong:** When the keyboard opens the app via URL scheme, the app briefly enters foreground then goes to background when user swipes back. The `.background` handler can interfere.
**Why it happens:** iOS fires scenePhase changes rapidly during app transitions.
**How to avoid:** The `isRecordingActive` guard (already applied) handles this. Ensure the state machine also respects this pattern.
**Warning signs:** Cold start recording works initially but keyboard freezes after swipe-back.

### Pattern: State Machine Transition Guard (for #42)

```swift
// Source: Architecture recommendation based on codebase analysis
private func validateTransition(from: LiveActivityPhase, to: LiveActivityPhase) -> Bool {
    let validTransitions: [LiveActivityPhase: Set<LiveActivityPhase>] = [
        .idle: [.standby],
        .standby: [.recording, .idle],
        .recording: [.transcribing, .standby],  // standby = cancel
        .transcribing: [.ready, .failed],
        .ready: [.standby, .recording],  // recording = quick chain
        .failed: [.standby]
    ]
    return validTransitions[from]?.contains(to) ?? false
}
```
