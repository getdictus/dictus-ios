### Human Verification Required

#### 1. Waveform animation after extension process suspension

**Test:** Put the app in background, open the keyboard, begin a recording to confirm waveform works. Then switch to the iOS Settings app (this suspends the keyboard extension process), return to the original app + keyboard, and start a new recording.
**Expected:** Waveform bars animate visibly during the second recording, after the extension process was suspended and resumed.
**Why human:** iOS process suspension and CADisplayLink restart behaviour cannot be reproduced in static analysis. The renderTick mechanism and objectWillChange.send() are correctly placed in code, but whether they actually unblock a suspended TimelineView's animation loop requires on-device observation.

#### 2. Dynamic Island stays in sync across 5+ chained recordings

**Test:** From any keyboard or DI tap entry path, complete 5 consecutive recordings rapidly — start recording, stop after 2 seconds, immediately start the next recording before the 1s auto-dismiss fires. Observe the Dynamic Island throughout.
**Expected:** DI shows .recording during each recording, .transcribing while processing, .ready briefly after each transcription, then .recording again on the next dictation. Never flashes standby mid-recording.
**Why human:** The state machine guards and auto-dismiss phase checks are wired correctly, but validating the timing of concurrent Task execution across 5 chained recordings requires real device interaction.

#### 3. Stale transcription card clearing via DI tap path

**Test:** Complete a dictation by tapping the Dynamic Island (not the keyboard mic button). When RecordingView shows the transcription card, tap the DI again to start a second recording.
**Expected:** RecordingView opens without any transcription card from the previous session — the view is clean.
**Why human:** The DI tap path requires Dynamic Island hardware (iPhone 14 Pro or later) and cannot be tested in simulator reliably.
