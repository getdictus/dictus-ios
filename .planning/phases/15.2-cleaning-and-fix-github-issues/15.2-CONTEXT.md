### New: Waveform off-screen recovery (#45)

**Root cause confirmed by 3 device log sessions:** iOS does NOT render SwiftUI views for off-screen keyboard extensions. When the keyboard goes off-screen (cold start app-switch dance, DI tap, app switching), SwiftUI stops body evaluation entirely. readWaveformData() continues polling App Group, renderTick increments, but zero body: logs appear. When keyboard returns, the overlay is stale/frozen.

**State recovery strategy:**
- Full state re-read from App Group on every keyboardDidAppear — not incremental, not conditional
- Read ALL state: dictationStatus, waveformEnergy, recordingElapsed, lastResult, bufferSeconds
- This replaces the current partial refreshFromDefaults() which doesn't handle all overlay-related state

**Waveform rendering approach:**
- Force BrandWaveform view recreation via .id() modifier on keyboardDidAppear
- Incrementing a waveformRefreshID in KeyboardState forces SwiftUI to destroy and recreate TimelineView with a fresh CADisplayLink
- Seed displayLevels from current energyLevels on .onAppear (so first frame shows data, not blank)

**Overlay lifecycle on reappear:**
- Re-evaluate overlay visibility on keyboardDidAppear based on CURRENT App Group status
- If status changed to idle/ready while keyboard was off-screen, overlay must hide IMMEDIATELY on reappear
- If status is still recording, overlay must show with CURRENT waveform data, not stale snapshot

**Timer/elapsed recovery:**
- Recalculate elapsed time from App Group recording start timestamp on reappear
- Do NOT rely on local Timer that stops when keyboard is off-screen
- Single source of truth: App Group timestamp, recalculated on every keyboardDidAppear

**Testing matrix (manual, on-device):**
1. Warm start — tap mic, record, transcribe (baseline, must not regress)
2. Cold start (app killed) — mic tap, app-switch dance, return to keyboard
3. Cold start (DI/notification stop) — stop from DI, then new recording
4. App switching during recording — Notes → ChatGPT → Safari → back to Notes
5. 5+ chained recordings — DI must stay in sync throughout
6. DI tap to start recording — overlay and waveform must work

### Claude's Discretion

- Whether to use DispatchSourceTimer or Timer.publish for off-screen-resilient timing
- Log instrumentation approach for verification
- Whether waveformRefreshID lives in KeyboardState or elsewhere

<specifics>
## Specific Ideas

- The fix must work at the **keyboard state management level**, not the SwiftUI rendering level — more @State tricks won't help when SwiftUI skips body evaluation entirely
- The recovery must handle 3 scenarios: (a) recording still active when keyboard returns, (b) recording finished while off-screen, (c) recording failed while off-screen
- Phase 12 lesson is critical: don't instant-reset on keyboardAppear — the cold start URL scheme causes rapid disappear/appear. The recovery must be additive (refresh state) not destructive (reset state)
- User specifically said "no more band-aids" — the fix must be architecturally sound
