**KeyboardState.swift -- add startRecording() method:**

Add a new public method `startRecording()` that:
1. Calls existing `markRequested()` (which writes `.requested` status to App Group and fires haptic)
2. Opens the `dictus://dictate` URL to trigger recording in DictusApp

WHY KeyboardState needs to open the URL: The actual WhisperKit recording still runs in DictusApp (keyboard extension has 50MB memory limit, cannot load ML models). The difference from Phase 2 is that the keyboard extension now controls the flow: it sets local state FIRST (so the recording overlay appears immediately), then signals the app to start recording. Previously the Link opened the app and the user stayed in the app.

To open a URL from keyboard extension without UIApplication.shared, use:
```swift
func startRecording() {
    markRequested()
    // Open DictusApp to begin recording.
    // WHY this works in extension: UIInputViewController inherits from UIViewController
    // which has access to the responder chain. We use the controller's extensionContext
    // or the openURL method on the parent view's environment.
    // Actually, the simplest approach: store the openURL environment action.
    // But KeyboardState is not a View. Instead, accept a URL-opening closure.
}
```

REVISED APPROACH: Since KeyboardState is not a SwiftUI View and cannot access `@Environment(\.openURL)`, the cleanest pattern is:
- Add a `var openURL: ((URL) -> Void)?` property to KeyboardState
- In `startRecording()`, call `markRequested()` then `openURL?(URL(string: "dictus://dictate")!)`
- KeyboardRootView sets `state.openURL` in `.onAppear` using the `@Environment(\.openURL)` action

This preserves the existing pattern where KeyboardRootView configures KeyboardState in `.onAppear` (same as how `state.controller` is set).
