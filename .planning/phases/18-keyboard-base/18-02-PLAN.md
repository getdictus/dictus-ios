  <action>
    Create DictusKeyboard/DictusKeyboardBridge.swift -- a class that implements the giellakbd-ios KeyboardView delegate protocol and translates key events into Dictus actions.

    The bridge must:

    1. **Implement the delegate protocol** from vendored KeyboardView.swift (read the exact protocol name and methods from the vendored file -- likely KeyboardViewDelegate or similar).

    2. **Hold a weak reference to UIInputViewController** for textDocumentProxy access:
       ```swift
       weak var controller: UIInputViewController?
       ```

    3. **Handle key actions by key type** (read exact KeyType enum from vendored KeyDefinition.swift):
       - **Input keys (character):**
         - Call `HapticFeedback.keyTapped()` (touchDown feedback)
         - Call `AudioServicesPlaySystemSound(1104)` for letter sound (if sound enabled)
         - Call `controller?.textDocumentProxy.insertText(character)`
         - Post `Notification.Name.dictusTextDidChange` for autocap check

       - **Backspace/delete:**
         - Call `HapticFeedback.keyTapped()`
         - Call `AudioServicesPlaySystemSound(1155)` for delete sound
         - Call `controller?.textDocumentProxy.deleteBackward()`
         - Post `.dictusTextDidChange`

       - **Spacebar:**
         - Call `HapticFeedback.keyTapped()`
         - Call `AudioServicesPlaySystemSound(1156)` for modifier sound
         - Call `handleAutoFullStop()` BEFORE inserting space (checks for double-space -> period)
         - Call `controller?.textDocumentProxy.insertText(" ")`
         - Post `.dictusTextDidChange`

       - **Return:**
         - Call `HapticFeedback.keyTapped()`
         - Call `AudioServicesPlaySystemSound(1156)` for modifier sound
         - Call `controller?.textDocumentProxy.insertText("\n")`
         - Post `.dictusTextDidChange`

       - **Shift (single tap):**
         - Call `HapticFeedback.keyTapped()`
         - Call `AudioServicesPlaySystemSound(1156)`
         - Toggle keyboard page: .normal -> .shifted, .shifted -> .normal, .capslock -> .normal

       - **Shift (double tap for caps lock):**
         - Call `HapticFeedback.keyTapped()`
         - Set keyboard page to .capslock
         - Double-tap detection: if shift tapped within 300ms of previous shift tap, activate caps lock

       - **Layer switch (123/ABC):**
         - Call `HapticFeedback.keyTapped()`
         - Call `AudioServicesPlaySystemSound(1156)`
         - Call `keyboardView.page.alternatePage()` to toggle between letters and symbols1

       - **Symbol toggle (#+=, 123 on symbols layer):**
         - Toggle between .symbols1 and .symbols2

    4. **Sound enabled check:** Read `AppGroup.defaults.object(forKey: SharedKeys.soundEnabled) as? Bool ?? true` and cache it. Only play AudioServicesPlaySystemSound if enabled. Refresh on keyboard appear.

    5. **Move KeySound enum** from current KeyboardView.swift into this bridge file (or a separate small file) since old KeyboardView.swift will be removed eventually.

    6. **handleAutoFullStop()** implementation (from giellakbd-ios, proven):
       ```swift
       private func handleAutoFullStop() {
           guard let proxy = controller?.textDocumentProxy,
                 let text = proxy.documentContextBeforeInput?.suffix(3),
                 text.count == 3,
                 text.suffix(2) == "  " else { return }
           let first = text.prefix(1)
           if first != "." && first != " " {
               proxy.deleteBackward()
               proxy.deleteBackward()
               proxy.insertText(". ")
           }
       }
       ```

    7. Add DictusKeyboardBridge.swift to DictusKeyboard target in project.pbxproj.

  <action>
    Rewrite KeyboardViewController to use a HYBRID approach: UIKit giellakbd-ios KeyboardView as a direct subview, plus SwiftUI UIHostingController for toolbar/overlay only.

    **Architecture (per RESEARCH.md Pitfall 6 recommendation):**
    The UIKit KeyboardView is added as a direct sibling subview of kbInputView, NOT wrapped in UIViewRepresentable inside SwiftUI. This avoids SwiftUI recreation issues. The SwiftUI hosting controller handles ONLY the toolbar and recording overlay.

    **KeyboardViewController changes:**

    1. **New properties:**
       ```swift
       private var keyboardView: KeyboardView?  // giellakbd-ios UICollectionView keyboard
       private var bridge: DictusKeyboardBridge? // delegate adapter
       ```

    2. **viewDidLoad() rewrite:**
       a. Create KeyboardInputView (kbInputView) same as before
       b. Create the giellakbd-ios KeyboardView:
          ```swift
          let definition = FrenchKeyboardLayouts.current()
          let theme = Theme.current  // or ThemeFactory.create() -- read vendored Theme.swift for exact API
          let giellaKeyboard = KeyboardView(definition: definition, theme: theme)
          ```
       c. Create DictusKeyboardBridge and wire it:
          ```swift
          let bridge = DictusKeyboardBridge()
          bridge.controller = self
          giellaKeyboard.delegate = bridge
          self.bridge = bridge
          self.keyboardView = giellaKeyboard
          ```
       d. Create SwiftUI hosting for toolbar+overlay ONLY. Modify KeyboardRootView to NOT include KeyboardView -- it should render ToolbarView + RecordingOverlay only (when recording). Create a new "KeyboardChromeView" or modify KeyboardRootView to exclude the keyboard grid.
       e. Add both views to kbInputView:
          ```swift
          // SwiftUI chrome (toolbar + recording overlay) at top
          kbInputView.addSubview(hostingController.view)
          // UIKit keyboard below
          kbInputView.addSubview(giellaKeyboard)
          ```
       f. Set up Auto Layout constraints:
          - Hosting view (toolbar): pinned to top, leading, trailing of kbInputView. Fixed height = toolbarHeight (52pt).
          - Keyboard view: pinned below hosting view, leading, trailing, bottom of kbInputView.
          - When recording overlay is active, hide the keyboard view and show full-height overlay in the hosting view.

    3. **Height calculation update:**
       Use giellakbd-ios KeyboardHeightProvider for the key grid height instead of the old KeyMetrics calculation:
       ```swift
       private func computeKeyboardHeight() -> CGFloat {
           let keyGridHeight = KeyboardHeightProvider.keyboardHeight(for: DeviceContext.current)
           let toolbarHeight: CGFloat = 52
           let bottomPadding: CGFloat = 8
           return keyGridHeight + toolbarHeight + bottomPadding
       }
       ```

    4. **Recording overlay handling:**
       When `KeyboardState.shared.dictationStatus` changes to recording/transcribing:
       - Hide `keyboardView` (giellaKeyboard.isHidden = true)
       - The SwiftUI hosting view expands or the recording overlay covers the keyboard area
       When recording ends:
       - Show `keyboardView` (giellaKeyboard.isHidden = false)

    5. **Keep existing lifecycle methods:**
       - viewWillAppear: keep PersistentLog, KeyboardState registration, height recalculation, cold start detection
       - viewDidDisappear: keep logging and state cleanup
       - textDidChange: keep notification posting
       - deinit: keep logging

    **KeyboardRootView changes:**

    1. Remove the `KeyboardView(...)` instantiation from the body. The SwiftUI view should now contain:
       - ToolbarView (always visible)
       - RecordingOverlay (when recording)
       - NO KeyboardView, NO SuggestionBarView (Phase 20), NO EmojiPickerView (Phase 20)

    2. The `keyboardHeight` computed property and `totalContentHeight` now refer to just the toolbar height for SwiftUI layout purposes. The UIKit keyboard handles its own height via Auto Layout.

    3. Keep all existing state management (@ObservedObject state, @ObservedObject waveformDriver, etc.) for recording overlay functionality.

    4. The SwiftUI body becomes approximately:
       ```swift
       var body: some View {
           VStack(spacing: 0) {
               ToolbarView(...)
               if showsOverlay {
                   RecordingOverlay(...)
               }
               // No keyboard here -- it's UIKit, managed by KeyboardViewController
           }
       }
       ```
