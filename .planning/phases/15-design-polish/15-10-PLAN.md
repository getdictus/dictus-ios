<task type="auto">
  <name>Task 1: Unify waveform vertical position across overlay states</name>
  <files>DictusKeyboard/Views/RecordingOverlay.swift</files>
  <action>
The root cause: `recordingContent` has a footer (timer + "Listening..." text, ~40pt) that pushes the waveform up inside the GeometryReader. `transcribingContent` uses symmetric Spacers with no footer, so the waveform sits at a different Y position. When transitioning from recording to transcribing, the waveform visually jumps up.

Fix by restructuring `transcribingContent` to match the vertical layout of `recordingContent`:

1. Change "Listening..." to "En écoute..." at line 161.

2. Restructure `transcribingContent` to use the SAME layout structure as `recordingContent`:
   - Top bar area (same height reserved, even if no buttons shown)
   - GeometryReader with waveform centered
   - Footer area with matching height (use "Transcription..." text in the same position as the timer + status text)

Replace the current `transcribingContent`:
```swift
private var transcribingContent: some View {
    VStack(spacing: 16) {
        Spacer()
        BrandWaveform(maxHeight: 60, isProcessing: true)
            .padding(.horizontal, 16)
        Text("Transcription...")
            .font(.dictusCaption)
            .foregroundColor(secondaryForeground)
        Spacer()
    }
}
```

With a layout that matches recordingContent's structure:
```swift
private var transcribingContent: some View {
    VStack(spacing: 0) {
        // Reserve same top bar height as recording state (buttons area)
        Color.clear
            .frame(height: 44)
            .padding(.horizontal, 16)
            .padding(.vertical, 8)

        // Waveform in GeometryReader — same structure as recording state
        GeometryReader { geo in
            VStack(spacing: 8) {
                Spacer(minLength: 0)

                BrandWaveform(maxHeight: geo.size.height * 0.7, isProcessing: true)
                    .padding(.horizontal, 2)

                Spacer(minLength: 0)
            }
            .frame(width: geo.size.width, height: geo.size.height)
        }

        // Footer — matches recording footer height (timer line + status line)
        Text("Transcription...")
            .font(.system(size: timerFontSize, weight: .medium, design: .monospaced))
            .foregroundColor(foregroundColor)
            .padding(.bottom, 4)

        Color.clear
            .frame(height: 16)
            .padding(.bottom, 8)
    }
}
```

Key insight: The footer in recording state has two elements (timer text at ~20pt font + "En écoute..." at caption font + padding). The transcribing footer must occupy the same total height. Using "Transcription..." as the main text at the same timerFontSize, plus a clear spacer matching the second line height, ensures the waveform's GeometryReader gets identical available space in both states.

Also update `requestedContent` for consistency: its footer already has "Démarrage..." text which is fine, but verify the structure matches (it uses GeometryReader already, which is good).

<task type="auto">
  <name>Task 2: Fix settings tap feedback to cover full row on all controls</name>
  <files>DictusApp/Views/SettingsView.swift</files>
  <action>
The root cause: `SettingsRowStyle` is a `ButtonStyle` which only affects `Button` controls. `NavigationLink`, `Link`, and other controls don't respond to `ButtonStyle`. Additionally, the background wraps the label content (text height), not the full row area.

Fix approach: Instead of a ButtonStyle, wrap each non-Button interactive row in a way that provides press feedback. The cleanest SwiftUI approach for List rows:

1. REMOVE the `SettingsRowStyle` struct entirely (lines 14-24).

2. REMOVE `.buttonStyle(SettingsRowStyle())` from the List (line 133).

3. For the NavigationLink and Link rows in the "A propos" section, the native iOS List already provides press highlight for NavigationLink by default. The issue is that `.scrollContentBackground(.hidden)` strips this.

The fix: Add `.listRowBackground()` with a pressable background to each row in the "A propos" section. Actually, since `.listRowBackground(Color.dictusAccent.opacity(0.05))` is already set per-section, this provides a static background.

Better approach: Use `.contentShape(Rectangle())` on each interactive row to ensure the full area is tappable, and rely on the fact that NavigationLink and Link already have built-in press feedback in List context. The real issue was that SettingsRowStyle was MASKING the native behavior for Buttons, and NavigationLink/Link were never getting ANY custom style.

Simplest correct fix:
- Remove SettingsRowStyle entirely
- Remove .buttonStyle(SettingsRowStyle())
- The native List press highlight will work for NavigationLink rows automatically
- For the Link row (GitHub), wrap it in a Button that opens the URL, so it gets native Button press highlight in the List context
- For the "Exporter les logs" Button, it already gets native press highlight without the custom ButtonStyle masking it

Change the GitHub Link (lines 92-100) from:
```swift
Link(destination: URL(string: "https://github.com/Pivii/dictus")!) {
    HStack {
        Text("GitHub")
        Spacer()
        Image(systemName: "arrow.up.right")
            .font(.caption)
            .foregroundColor(.secondary)
    }
}
```
To a Button that opens the URL (so it gets native List row press highlight):
```swift
Button {
    UIApplication.shared.open(URL(string: "https://github.com/Pivii/dictus")!)
} label: {
    HStack {
        Text("GitHub")
        Spacer()
        Image(systemName: "arrow.up.right")
            .font(.caption)
            .foregroundColor(.secondary)
    }
}
```

This ensures ALL interactive rows in Settings are either Button or NavigationLink, both of which get native List row press highlight (the full-width gray flash) without needing a custom ButtonStyle.
