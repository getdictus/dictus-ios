# Phase 23.1: App Localization Audit & Fix - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Ensure all UI strings across DictusApp and DictusKeyboard use iOS String Catalogs for localization. The app UI must render in the user's device language (French or English, with English fallback). The in-app language picker is decoupled from UI language and controls only the transcription engine.

</domain>

<decisions>
## Implementation Decisions

### Localization mechanism
- Use iOS 17+ String Catalogs (.xcstrings), not classic Localizable.strings
- One String Catalog per target: DictusApp, DictusKeyboard, DictusCore (if shared strings exist)
- Vendored keyboard code (giellakbd-ios) is NOT touched — only Dictus-authored strings are localized

### Language switching
- UI language follows iOS device language automatically (standard iOS behavior via String Catalog localization selection)
- English is the fallback language (not French) — if device is set to an unsupported language, UI shows English
- The in-app Settings picker ("Langue") is renamed to "Transcription language" / "Langue de transcription" and controls ONLY the WhisperKit/Parakeet speech engine language — it no longer implies UI language control
- Remove any manual language-switching logic in views (e.g., the ternary `language == "fr" ? ... : ...` in SwipeBackOverlayView) — let String Catalogs handle it natively

### Keyboard extension scope
- Dictus-authored keyboard UI strings ARE localized: mic toolbar labels, recording overlay text, Full Access banner, suggestion bar labels, swipe-back overlay
- Keyboard UI language follows device language (same as app), independent of transcription language
- Vendored keyboard internals (key labels, giellakbd-ios strings) are NOT modified

### String coverage & development language
- All ~100+ hardcoded strings across ~23 files are extracted in one pass — no phased rollout
- English is the development language (source strings in code are English)
- All current French strings in code are replaced with English equivalents
- French translations are provided in the String Catalog fr localization
- No exceptions — all user-visible strings go through String Catalogs (AZERTY/QWERTY are universal terms but will still be in the catalog for consistency)

### Claude's Discretion
- String key naming convention (Xcode auto-generates keys from the English source text by default)
- How to handle string interpolation / dynamic values (e.g., model names, file sizes)
- Whether to add accessibility label localizations in this pass or defer
- Grouping/commenting strategy within String Catalog files

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `SharedKeys.language` — Already stores "fr"/"en" for transcription; will remain but be clearly scoped to transcription only
- `AppGroup.defaults` — Shared UserDefaults for cross-process preference access

### Established Patterns
- SwiftUI `Text("...")` is used throughout DictusApp — these auto-localize with String Catalogs
- UIKit code in keyboard uses direct string literals — needs explicit `NSLocalizedString()` or `String(localized:)` for localization
- SwipeBackOverlayView has manual `language == "fr" ? ... : ...` ternaries — pattern to be REMOVED and replaced by native String Catalog resolution

### Integration Points
- `SettingsView.swift` — Language picker needs label rename and scope clarification
- `DictusApp.swift` — Default language initialization may need adjustment (currently sets "fr" as default for SharedKeys.language)
- Info.plist files — May need `CFBundleLocalizations` array with ["en", "fr"]
- Xcode project — String Catalog files need to be added to correct targets

</code_context>
