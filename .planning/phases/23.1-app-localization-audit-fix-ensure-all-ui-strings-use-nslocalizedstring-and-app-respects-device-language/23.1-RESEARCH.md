# Phase 23.1: App Localization Audit & Fix - Research

**Researched:** 2026-04-01
**Domain:** iOS String Catalogs localization (SwiftUI + UIKit keyboard extension)
**Confidence:** HIGH

## Summary

Dictus currently has ~100 hardcoded French strings across 22+ files in DictusApp and DictusKeyboard targets, with zero localization infrastructure (no .xcstrings, no .strings, no .lproj directories). The Xcode project's `developmentRegion` is set to `fr` and `knownRegions` only contains `(fr, Base)` -- both need to change to `en` as the development language per user decision.

iOS 17+ String Catalogs (.xcstrings) are the correct mechanism. SwiftUI `Text("literal string")` auto-localizes via `LocalizedStringKey` -- no code changes needed for these once the catalog exists and the source strings are in English. The keyboard extension (SwiftUI views) works identically since each target gets its own String Catalog with its own bundle. The main work is: (1) change developmentRegion to en, (2) replace all French string literals with English equivalents, (3) create String Catalogs per target, (4) provide French translations in the catalogs, (5) rename the language picker label.

**Primary recommendation:** Create one `Localizable.xcstrings` per target (DictusApp, DictusKeyboard). Replace all French string literals in code with English. Build to auto-populate catalogs, then add French translations. Change `developmentRegion` to `en` in project.pbxproj.

## Standard Stack

### Core
| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| Xcode String Catalogs (.xcstrings) | Format 1.0 (Xcode 15+) | Centralized localization management | Apple's recommended replacement for .strings/.stringsdict; auto-extracts from SwiftUI Text; backward compatible at build time |
| SwiftUI LocalizedStringKey | iOS 17+ | Auto-localization of Text views | Text("literal") auto-looks up String Catalog; zero-code localization for SwiftUI |
| String(localized:) | iOS 16+ | Explicit localized string in Swift | Modern replacement for NSLocalizedString; works with String Catalogs |

### Supporting
| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| NSLocalizedString() | Foundation | Legacy localization API | Only if UIKit code needs explicit localization (vendored code -- NOT in scope) |
| Bundle.module | SPM | Package resource bundle | If DictusCore needs its own String Catalog |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| String Catalogs | Localizable.strings | Older format, no auto-extraction, no visual editor -- rejected per user decision |
| Auto-generated keys | Manual key naming (e.g., "settings.language.title") | Auto-generated keys from English text are simpler and Xcode default; manual keys add maintenance burden for ~100 strings |

## Architecture Patterns

### Recommended Project Structure
```
DictusApp/
  Localizable.xcstrings          # App target String Catalog (en + fr)
DictusKeyboard/
  Localizable.xcstrings          # Keyboard extension String Catalog (en + fr)
DictusCore/
  Sources/DictusCore/
    Resources/
      Localizable.xcstrings      # Only if DictusCore has user-visible strings
```

### Pattern 1: SwiftUI Text Auto-Localization
**What:** SwiftUI `Text("string literal")` automatically uses `LocalizedStringKey` and looks up the String Catalog
**When to use:** All SwiftUI views (both DictusApp and DictusKeyboard)
**Example:**
```swift
// BEFORE (French hardcoded):
Text("Modele vocal")

// AFTER (English source, French in String Catalog):
Text("Voice model")
// String Catalog: "Voice model" -> fr: "Modele vocal"
```

### Pattern 2: SwiftUI View Modifiers with Strings
**What:** `.navigationTitle()`, `Section()`, `Toggle()`, `Picker()`, `NavigationLink()`, `Label()`, `Button()` all accept `LocalizedStringKey`
**When to use:** All SwiftUI view modifiers that take string parameters
**Example:**
```swift
// BEFORE:
.navigationTitle("Reglages")
Section("Clavier") { ... }
Toggle("Retour haptique", isOn: $hapticsEnabled)

// AFTER:
.navigationTitle("Settings")
Section("Keyboard") { ... }
Toggle("Haptic feedback", isOn: $hapticsEnabled)
```

### Pattern 3: Dynamic String Variables (NOT auto-localized)
**What:** When a `String` variable is passed to `Text()`, SwiftUI uses verbatim display -- no localization lookup
**When to use:** Model descriptions, computed strings, interpolated values
**Example:**
```swift
// This does NOT auto-localize (model.description is String, not literal):
Text(model.description)

// Option A: Use localization key in DictusCore model data
// model.descriptionKey = "model.description.fast_but_inaccurate"
// Then: Text(LocalizedStringKey(model.descriptionKey))

// Option B (RECOMMENDED): Use String(localized:) at the display site
// Keep model.description as English source text
// Text(String(localized: String.LocalizationValue(model.description)))
// -- This won't work because String Catalog needs compile-time keys

// Option C (SIMPLEST): Replace description with a localizable computed property
// in the view layer, using a switch on model.identifier
```

### Pattern 4: String Interpolation in String Catalogs
**What:** SwiftUI handles interpolation automatically in String Catalogs
**When to use:** Dynamic values like percentages, model names
**Example:**
```swift
// This works -- Xcode extracts the pattern:
Text("\(Int(progress * 100))%")
// String Catalog key: "%lld%" -> fr: "%lld %"

// For named parameters:
Text("Delete \(model.displayName)?")
// String Catalog key: "Delete %@?" -> fr: "Supprimer %@ ?"
```

### Pattern 5: Keyboard Extension Bundle
**What:** Keyboard extension has its own bundle; String Catalog must be in the extension target
**When to use:** All DictusKeyboard localized strings
**Key insight:** `Bundle.main` in a keyboard extension refers to the extension's bundle, NOT the app's bundle. So `Text("literal")` in keyboard SwiftUI views looks up the keyboard's own `Localizable.xcstrings` automatically. No special bundle parameter needed.

### Anti-Patterns to Avoid
- **Manual language switching:** `language == "fr" ? "French text" : "English text"` -- let String Catalogs handle this natively
- **Shared String Catalog across targets:** Each target must have its own catalog; don't try to share one between app and extension
- **Text(verbatim:) for localizable strings:** Only use `Text(verbatim:)` for strings that should NOT be localized (URLs, code, technical identifiers)
- **Forgetting to build:** Xcode only populates String Catalogs on build -- new strings won't appear until you build

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Language detection | Manual `Locale.current.language` checks | String Catalog auto-resolution | iOS handles device language matching automatically |
| Pluralization | if count == 1 / else | String Catalog "Vary by Plural" | Handles all CLDR plural categories (zero, one, two, few, many, other) |
| String file management | Manual .strings files | .xcstrings visual editor | Auto-extraction, visual completion tracking, JSON format |
| Language fallback | Manual fallback logic | Xcode localization settings | Development language is automatic fallback |
| String key management | Enum-based key system | Auto-generated keys from English source text | Xcode default behavior, zero maintenance overhead |

## Common Pitfalls

### Pitfall 1: developmentRegion Still Set to "fr"
**What goes wrong:** If developmentRegion remains "fr", Xcode treats French as the source language and English as a translation. The fallback language for unsupported locales would be French, not English.
**Why it happens:** The Xcode project was created with French as the development language.
**How to avoid:** Change `developmentRegion = fr;` to `developmentRegion = en;` in project.pbxproj. Update `knownRegions` to `(en, fr, Base)`. Verify `DEVELOPMENT_LANGUAGE` build setting resolves to "en".
**Warning signs:** String Catalog shows French as "Source Language" instead of English.

### Pitfall 2: Text(stringVariable) Does Not Auto-Localize
**What goes wrong:** `Text(model.description)` where description is a `String` property displays verbatim -- never looks up the String Catalog.
**Why it happens:** SwiftUI `Text` has two initializers: `Text(_ key: LocalizedStringKey)` for literals and `Text(verbatim: String)`. When you pass a `String` variable, Swift resolves to verbatim.
**How to avoid:** For strings that come from data models (ModelInfo.description), either: (a) use `Text(LocalizedStringKey(stringVariable))` if the variable contains a known catalog key, or (b) handle localization at the model layer using `String(localized:)`.
**Warning signs:** Model descriptions always show in one language regardless of device setting.

### Pitfall 3: String Catalog Not Added to Correct Target
**What goes wrong:** Strings work in the app but not in the keyboard extension (or vice versa).
**Why it happens:** The .xcstrings file's target membership is set to only one target.
**How to avoid:** Create separate Localizable.xcstrings files -- one per target. Verify target membership in Xcode File Inspector. Each target needs its own catalog because they have different bundles.
**Warning signs:** Strings appear as raw English keys in the keyboard but translated in the app.

### Pitfall 4: Forgetting to Build After Adding Strings
**What goes wrong:** New strings don't appear in the String Catalog editor.
**Why it happens:** Xcode only scans and extracts localizable strings during the build phase.
**How to avoid:** Build the project (Cmd+B) after changing string literals. Check the String Catalog editor to verify new strings appeared.
**Warning signs:** String Catalog shows fewer entries than expected, "Stale" markers.

### Pitfall 5: String Interpolation Key Mismatch
**What goes wrong:** Translated strings with interpolation show wrong format or crash.
**Why it happens:** The format specifiers in the French translation don't match the English source.
**How to avoid:** When adding French translations for interpolated strings, preserve the same `%@`, `%lld`, etc. format specifiers in the same order.
**Warning signs:** Runtime crashes or garbled output in translated strings with dynamic values.

### Pitfall 6: DictusCore Model Descriptions Are Data, Not UI
**What goes wrong:** ModelInfo.description strings ("Rapide mais imprecis") are never localized because they're `String` properties in a Swift Package, not `Text()` literals.
**Why it happens:** These are static data defined in DictusCore, consumed as `String` in views.
**How to avoid:** Decision needed (Claude's discretion): either (a) make descriptions localization keys and use `String(localized:)` in the view, or (b) add a `localizedDescription` computed property in the app target that maps identifiers to localized strings. Option (b) is simpler -- keep DictusCore data-only, localize at the view layer.
**Recommendation:** Use option (b) -- add a small extension or helper in DictusApp that maps `ModelInfo.identifier` to a localized description string. This avoids making DictusCore depend on localization infrastructure.

## Code Examples

### Creating a String Catalog in Xcode
```
1. File > New > File > String Catalog
2. Name it "Localizable" (default)
3. Add to target: DictusApp (or DictusKeyboard)
4. Build the project (Cmd+B)
5. Open Localizable.xcstrings -- Xcode auto-populated English strings
6. Click "+" to add French localization
7. Fill in French translations for each key
```

### Changing developmentRegion in project.pbxproj
```
// BEFORE:
developmentRegion = fr;
knownRegions = (
    fr,
    Base,
);

// AFTER:
developmentRegion = en;
knownRegions = (
    en,
    fr,
    Base,
);
```

### Renaming Language Picker
```swift
// BEFORE:
Section {
    Picker("Langue", selection: $language) {
        Text("Francais").tag("fr")
        Text("English").tag("en")
    }
} header: {
    Text("Transcription")
}

// AFTER:
Section {
    Picker("Transcription language", selection: $language) {
        Text("Francais").tag("fr")
        Text("English").tag("en")
    }
} header: {
    Text("Transcription")
}
// String Catalog: "Transcription language" -> fr: "Langue de transcription"
// Note: "Francais" and "English" are language NAMES -- they stay as-is (not translated)
```

### Removing Manual Language Switching
```swift
// BEFORE (SwipeBackOverlayView.swift):
@AppStorage(SharedKeys.language, store: AppGroup.defaults)
private var language = "fr"

Text(language == "fr"
     ? "Glisse pour revenir au clavier"
     : "Swipe back to the keyboard")

// AFTER:
// Remove @AppStorage language property entirely
Text("Swipe back to the keyboard")
// String Catalog: "Swipe back to the keyboard" -> fr: "Glisse pour revenir au clavier"
```

### Localizing Model Descriptions at View Layer
```swift
// DictusApp/Extensions/ModelInfo+Localized.swift
import DictusCore
import SwiftUI

extension ModelInfo {
    /// Localized description for display in model cards.
    /// Source strings are in English; French provided via String Catalog.
    var localizedDescription: String {
        switch identifier {
        case "openai_whisper-tiny", "openai_whisper-base":
            return String(localized: "Fast but inaccurate")
        case "openai_whisper-small":
            return String(localized: "Accurate and balanced")
        case "openai_whisper-small_216MB":
            return String(localized: "Compact and fast")
        case "openai_whisper-medium":
            return String(localized: "Best accuracy")
        case "parakeet-tdt-0.6b-v3":
            return String(localized: "Fast and accurate (NVIDIA)")
        default:
            return description // Fallback to raw description
        }
    }
}
```

## Open Questions

1. **DictusCore String Catalog: needed or not?**
   - What we know: DictusCore has model descriptions in French ("Rapide mais imprecis") and displayName strings. displayName values ("Tiny", "Base", "Small") are technical names that don't need translation. Descriptions do.
   - What's unclear: Whether to add localization infrastructure to the Swift Package or handle at view layer.
   - Recommendation: Handle at view layer (ModelInfo+Localized extension). Avoids Package.swift changes and keeps DictusCore as a pure data/logic layer. Change DictusCore descriptions to English to match development language convention.

2. **Accessibility labels: this pass or defer?**
   - What we know: Only 1 accessibility label found (`accessibilityLabel("Arreter l'enregistrement")`). Accessibility modifiers accept `LocalizedStringKey` and auto-localize.
   - Recommendation: Include in this pass -- it's 1 string. Deferring creates unnecessary follow-up work.

3. **EmojiSearchFR: localization scope?**
   - What we know: `EmojiSearchFR.swift` has a French keyword dictionary for emoji search. The `language == "fr"` check in EmojiPickerView line 307 switches search behavior.
   - Recommendation: Emoji search keywords are data/functionality, not UI strings. The language-conditional search logic is legitimate (searching emojis by French keywords when transcription language is French). Keep as-is -- this is NOT a localization concern.
