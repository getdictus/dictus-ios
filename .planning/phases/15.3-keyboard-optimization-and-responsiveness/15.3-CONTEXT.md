# Phase 15.3: Keyboard Optimization and Responsiveness - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Reduce keyboard input latency to match native iOS keyboard feel, and adjust key spacing/dimensions to match Apple keyboard proportions. Covers GitHub issues #44 (keyboard responsiveness) and #29 (key spacing and dimensions).

</domain>

<decisions>
## Implementation Decisions

### Touch pipeline
- Replace SwiftUI DragGesture with UIKit UIViewRepresentable (touchesBegan/touchesEnded) for **letter keys only**
- Keep DragGesture for keys that need continuous tracking: accent long-press, delete repeat, space trackpad
- Audio AND haptic fire on **touchDown** (not touchUp) — matches Apple's native keyboard behavior (user tested and confirmed)
- Call `HapticFeedback.prepare()` on every touchDown, even during rapid typing
- Pipeline order on touchDown: visual highlight -> audio -> haptic -> prepare() for next
- Pipeline order on touchUp: cancel long-press timer -> insertText -> update suggestions (async)
- Move `SuggestionState.update()` to background thread with coalescing (cancel in-flight on new keystroke)
- Cache `HapticFeedback.isEnabled()` in a static var instead of reading UserDefaults on every tap

### Key dimensions (#29)
- Full review of ALL spacing values: side margins, inter-key horizontal gap, inter-row vertical gap, corner radius
- Research approach priority: (1) find published Apple keyboard values, (2) measure from screenshots if not found, (3) visual tuning as last resort
- Device-adaptive metrics: 3 device classes (SE/compact, standard, Plus/Max) with different spacing/margin values — same approach as existing keyHeight adaptation
- Make `KeyMetrics.keyHeight` a static let (computed once) instead of computed var (recalculated every access)

### Visual press feedback (per key type, matching Apple exactly)
- **Letter keys + accent keys**: popup only, NO background color change (current behavior is correct)
- **Delete, Space, Return keys**: background color change on press (brighter in dark mode, darker in light mode)
- **Symbol toggle key**: label transforms ("123" <-> "ABC") — already implemented
- **Shift key**: icon state changes (empty -> filled -> lock) — already implemented
- Keep existing KeyPopup preview above pressed letter keys

### Measurement & instrumentation
- Add OSSignposter signposts on key tap pipeline (touchDown->highlight, touchDown->haptic, touchUp->insertText)
- Measure baseline on physical device via Xcode Instruments (Product > Profile) BEFORE optimization
- Measure again AFTER optimization to verify improvement
- Target latencies from #44: T1 (touchDown->highlight) <= 16.67ms, T2 (touchDown->haptic) <= 16.67ms, T3 (touchUp->insertText) <= 33ms

### Claude's Discretion
- Exact implementation of UIViewRepresentable touch handler for letter keys
- Background thread queue configuration for suggestions (serial vs concurrent, QoS level)
- How to structure the device class enum for adaptive metrics
- Whether to use `.drawingGroup()` or `Equatable` conformance for SwiftUI render optimization
- Exact color values for press feedback on delete/space/return keys

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `HapticFeedback` enum with pre-allocated static generators — extend with `prepareForNextTap()` on touchDown
- `KeyMetrics` enum — refactor to device-adaptive with cached values
- `KeySound` system sound IDs — already correct (letter/delete/modifier), just needs reordering in pipeline
- `AccentPopup` and `AccentedCharacters` — accent long-press logic stays on DragGesture, not affected by UIKit migration

### Established Patterns
- `DictusCore` shared framework for cross-target code (HapticFeedback, AppGroup, SharedKeys)
- `#if canImport(UIKit) && !os(macOS)` guards in DictusCore for macOS test compatibility
- Pre-allocated static instances pattern (haptic generators) — extend to cached UserDefaults values
- `dictusGlass` modifier for iOS 26 Liquid Glass — key backgrounds use this

### Integration Points
- `KeyButton` is used by `KeyRow.keyView(for:)` for `.character` type keys — UIKit replacement targets this case
- `KeyboardView.insertCharacter()` is the main insertion pipeline — audio/haptic reordering happens here
- `SuggestionState.update(proxy:)` called after every keystroke — move to background queue
- `KeyboardRootView.onAppear` calls `HapticFeedback.warmUp()` — already correct initialization point

</code_context>
