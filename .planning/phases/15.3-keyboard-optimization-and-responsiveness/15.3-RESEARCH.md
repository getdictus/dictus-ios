# Phase 15.3: Keyboard Optimization and Responsiveness - Research

**Researched:** 2026-03-22
**Domain:** iOS keyboard extension performance (touch pipeline, rendering, layout, haptics)
**Confidence:** HIGH

## Summary

This phase addresses two GitHub issues: #44 (keyboard responsiveness / input latency) and #29 (key spacing / dimensions). The CONTEXT.md provides exceptionally detailed locked decisions covering the touch pipeline rewrite (UIViewRepresentable for letter keys), haptic/audio timing (fire on touchDown), suggestion offloading to background thread, key dimension research approach, and per-key-type visual press feedback.

The existing codebase uses SwiftUI DragGesture for all keys, fires haptics on touchUp (not touchDown), reads UserDefaults on every haptic call, computes suggestions on main thread, and has a computed `keyHeight` property that recalculates on every access. All of these are identified bottlenecks in the issue analysis.

The existing deep research document (`.planning/research/keyboard-responsiveness.md`) provides the theoretical foundation. This research supplements it with implementation-specific findings for the UIViewRepresentable pattern, OSSignposter instrumentation, Apple keyboard metrics, and SwiftUI render optimization.

**Primary recommendation:** Implement in 3 waves: (1) instrumentation + measurement baseline, (2) quick wins (haptic caching, pipeline reorder, KeyMetrics static let, suggestion background queue), (3) UIViewRepresentable touch handler for letter keys + key dimension overhaul.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Replace SwiftUI DragGesture with UIKit UIViewRepresentable (touchesBegan/touchesEnded) for **letter keys only**
- Keep DragGesture for keys that need continuous tracking: accent long-press, delete repeat, space trackpad
- Audio AND haptic fire on **touchDown** (not touchUp) -- matches Apple's native keyboard behavior (user tested and confirmed)
- Call `HapticFeedback.prepare()` on every touchDown, even during rapid typing
- Pipeline order on touchDown: visual highlight -> audio -> haptic -> prepare() for next
- Pipeline order on touchUp: cancel long-press timer -> insertText -> update suggestions (async)
- Move `SuggestionState.update()` to background thread with coalescing (cancel in-flight on new keystroke)
- Cache `HapticFeedback.isEnabled()` in a static var instead of reading UserDefaults on every tap
- Full review of ALL spacing values: side margins, inter-key horizontal gap, inter-row vertical gap, corner radius
- Research approach priority for key dimensions: (1) find published Apple keyboard values, (2) measure from screenshots if not found, (3) visual tuning as last resort
- Device-adaptive metrics: 3 device classes (SE/compact, standard, Plus/Max) with different spacing/margin values
- Make `KeyMetrics.keyHeight` a static let (computed once) instead of computed var
- Letter keys + accent keys: popup only, NO background color change
- Delete, Space, Return keys: background color change on press (brighter in dark mode, darker in light mode)
- Add OSSignposter signposts on key tap pipeline
- Measure baseline on physical device via Instruments BEFORE optimization
- Target latencies: T1 (touchDown->highlight) <= 16.67ms, T2 (touchDown->haptic) <= 16.67ms, T3 (touchUp->insertText) <= 33ms

### Pattern 1: UIViewRepresentable Touch Handler for Letter Keys
**What:** A UIView subclass wrapped in UIViewRepresentable that overrides touchesBegan/touchesEnded for zero-overhead touch detection. The UIView communicates touch events back to SwiftUI via closures.
**When to use:** Letter keys only (`.character` type in KeyRow).
**Why:** SwiftUI DragGesture adds gesture resolution overhead (conflict resolution, minimum distance checks). UIKit touch events fire immediately on the UIResponder chain with no gesture disambiguation delay.

```swift
// Source: UIKit UIResponder documentation + UIViewRepresentable Apple docs
import SwiftUI
import UIKit

struct LetterKeyTouchView: UIViewRepresentable {
    let label: String
    let onTouchDown: () -> Void
    let onTouchUp: () -> Void
    let onLongPress: () -> Void  // 400ms timer fires

    func makeUIView(context: Context) -> LetterKeyUIView {
        let view = LetterKeyUIView()
        view.onTouchDown = onTouchDown
        view.onTouchUp = onTouchUp
        view.onLongPress = onLongPress
        return view
    }

    func updateUIView(_ uiView: LetterKeyUIView, context: Context) {
        // Update label if needed
    }
}

final class LetterKeyUIView: UIView {
    var onTouchDown: (() -> Void)?
    var onTouchUp: (() -> Void)?
    var onLongPress: (() -> Void)?
    private var longPressTimer: Task<Void, Never>?

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        super.touchesBegan(touches, with: event)
        onTouchDown?()
        // Start 400ms timer for accent long-press
        longPressTimer?.cancel()
        longPressTimer = Task { @MainActor in
            try? await Task.sleep(nanoseconds: 400_000_000)
            guard !Task.isCancelled else { return }
            onLongPress?()
        }
    }

    override func touchesEnded(_ touches: Set<UITouch>, with event: UIEvent?) {
        super.touchesEnded(touches, with: event)
        longPressTimer?.cancel()
        longPressTimer = nil
        onTouchUp?()
    }

    override func touchesCancelled(_ touches: Set<UITouch>, with event: UIEvent?) {
        super.touchesCancelled(touches, with: event)
        longPressTimer?.cancel()
        longPressTimer = nil
    }
}
```

**Important caveat:** When long-press triggers accent popup, the accent selection drag needs to still work. The UIView must communicate drag position back for accent selection. Consider: if long-press fires, switch to a state where touchesMoved tracks position for accent cell selection (same math as current `updateSelectedAccent`).

### Pattern 2: Background Suggestion Queue with Coalescing
**What:** Move `SuggestionState.update()` work off the main thread. Cancel in-flight computations when a new keystroke arrives.
**When to use:** Every keystroke triggers a suggestion update.

```swift
// Recommendation: Serial DispatchQueue with .userInitiated QoS
// Serial ensures ordering. .userInitiated because suggestions are user-facing
// but not blocking the keyboard interaction.
private let suggestionQueue = DispatchQueue(
    label: "com.pivi.dictus.suggestions",
    qos: .userInitiated
)
private var currentSuggestionWork: DispatchWorkItem?

func update(proxy: UITextDocumentProxy) {
    // Read proxy data on main thread (UIKit requirement)
    guard let context = proxy.documentContextBeforeInput, !context.isEmpty else {
        DispatchQueue.main.async { self.clear() }
        return
    }

    // Cancel previous in-flight computation
    currentSuggestionWork?.cancel()

    let work = DispatchWorkItem { [weak self] in
        guard let self = self else { return }
        // Compute suggestions on background
        let partial = self.extractLastWord(from: context)
        guard !partial.isEmpty else {
            DispatchQueue.main.async { self.clear() }
            return
        }

        let accents = self.engine.accentSuggestions(for: partial)
        let completions = accents == nil ? self.engine.suggestions(for: partial) : nil

        // Publish results on main thread
        DispatchQueue.main.async {
            guard !(self.currentSuggestionWork?.isCancelled ?? true) else { return }
            self.currentWord = partial
            if let accents = accents {
                self.suggestions = accents
                self.mode = .accents
            } else if let completions = completions, !completions.isEmpty {
                self.suggestions = completions
                self.mode = .completions
            } else {
                self.clear()
            }
        }
    }

    currentSuggestionWork = work
    suggestionQueue.async(execute: work)
}
```

### Pattern 4: Device-Adaptive KeyMetrics
**What:** Replace computed `keyHeight` var with a device class enum that precomputes all layout values once.
**When to use:** All key dimension access throughout the keyboard.

```swift
enum DeviceClass {
    case compact    // iPhone SE, mini (screen height <= 667pt)
    case standard   // iPhone 14/15/16 (screen height <= 852pt)
    case large      // iPhone Plus/Max (screen height > 852pt)

    static let current: DeviceClass = {
        let h = UIScreen.main.bounds.height
        if h <= 667 { return .compact }
        else if h <= 852 { return .standard }
        else { return .large }
    }()
}

enum KeyMetrics {
    // All values computed once at app launch
    static let keyHeight: CGFloat = {
        switch DeviceClass.current {
        case .compact:  return 42
        case .standard: return 46
        case .large:    return 50
        }
    }()

    static let rowSpacing: CGFloat = {
        switch DeviceClass.current {
        case .compact:  return 8
        case .standard: return 10
        case .large:    return 11
        }
    }()

    static let keySpacing: CGFloat = {
        switch DeviceClass.current {
        case .compact:  return 5
        case .standard: return 6
        case .large:    return 6
        }
    }()

    static let rowHorizontalPadding: CGFloat = {
        switch DeviceClass.current {
        case .compact:  return 3
        case .standard: return 3
        case .large:    return 4
        }
    }()

    static let keyCornerRadius: CGFloat = 5

    // ... letterKeyColor stays as-is (already static let) ...
}
```

**Note on exact values:** Apple does not publish official keyboard dimension specs in points. The values above are estimates. The CONTEXT.md decision says: research first, measure from screenshots second, visual tune third. Key insight from KeyboardKit (popular iOS keyboard framework): they use 51pt row height for iOS 26. Apple keyboard side margins are approximately equal to inter-key gap (issue #29 observation). Current Dictus: 3pt margins, 4pt key spacing -- margins need to increase to approximately match key spacing.

### Anti-Patterns to Avoid
- **Do not use .drawingGroup() on the keyboard view:** `drawingGroup()` renders to an off-screen Metal texture. For a keyboard where individual keys highlight independently, this would force re-rendering the entire keyboard texture on every keypress. Use it only if profiling shows SwiftUI compositing is the bottleneck, not individual key updates.
- **Do not make all views Equatable preemptively:** Only add Equatable conformance to views where profiling shows unnecessary re-renders. The `ForEach` over key rows should be stable with proper `id:` usage.
- **Do not use DispatchQueue.main.async for touch-down feedback:** The touchDown handler must execute synchronously on the main thread for zero-frame-delay visual/haptic/audio response.
- **Do not read UITextDocumentProxy on a background thread:** The proxy is a UIKit object bound to the main thread. Read the context string on main, then dispatch computation to background.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Touch event timing | Custom gesture recognizer | UIView touchesBegan/Ended | Direct UIResponder is the lowest-latency path |
| Performance measurement | print() timestamps | OSSignposter + Instruments | Sub-microsecond precision, visual timeline, no runtime overhead when not profiling |
| Haptic engine management | Custom timing logic | UIImpactFeedbackGenerator.prepare() | Apple's API handles Taptic Engine warmup internally |
| Work item cancellation | Manual Bool flags | DispatchWorkItem.cancel() | Thread-safe, built into GCD |
| Keyboard height | Hardcoded values | UIScreen.main.bounds-based computation | Device landscape changes across iPhone generations |

**Key insight:** The touch latency budget is 16.67ms (one frame at 60fps). Every unnecessary computation in the touch-down path directly adds to perceived lag. The pipeline must be: highlight -> audio -> haptic -> prepare. Everything else (insert, suggestions, autocorrect) happens on touchUp or async.

## Common Pitfalls

### Pitfall 1: UIViewRepresentable Gesture Conflict with SwiftUI
**What goes wrong:** When wrapping a UIView in UIViewRepresentable, SwiftUI may intercept touches before they reach the UIView, especially if parent views have gesture modifiers.
**Why it happens:** SwiftUI's hit testing and gesture system operates above UIKit's touch delivery.
**How to avoid:** Ensure no parent view of the letter key has a `.gesture()` or `.onTapGesture()` modifier that could swallow touches. The UIView should be the direct content of the representable, not nested under other gesture-bearing views.
**Warning signs:** Touch events fire intermittently or not at all on the UIView.

### Pitfall 2: Accent Long-Press Transition
**What goes wrong:** After replacing DragGesture with UIKit touches for letter keys, the accent popup long-press stops working because it relied on DragGesture's continuous position tracking.
**Why it happens:** UIView's touchesMoved provides raw CGPoint positions, not the SwiftUI DragGesture value with its location property.
**How to avoid:** Implement accent selection tracking in touchesMoved on the UIView. Communicate the drag position back to SwiftUI via a closure or Coordinator. The accent popup SwiftUI view can still render as an overlay, but the hit-testing math moves into the UIView.
**Warning signs:** Accent popup appears but cannot select different accents by sliding.

### Pitfall 3: HapticFeedback.isEnabled Cache Staleness
**What goes wrong:** User toggles haptics off in Settings, but keyboard keeps vibrating because the cached value is stale.
**Why it happens:** Static var is computed once and never refreshed.
**How to avoid:** Refresh the cache in `KeyboardRootView.onAppear` (which fires on every keyboard appearance) and when the keyboard receives a settings-changed notification. A `refreshCache()` static method is clean.
**Warning signs:** Haptic setting changes don't take effect until keyboard process restarts.

### Pitfall 4: Background Queue + @Published Thread Safety
**What goes wrong:** Publishing changes to `@Published` properties from a background queue causes runtime warnings ("Publishing changes from background threads is not allowed").
**Why it happens:** `SuggestionState` is an ObservableObject with @Published properties. SwiftUI requires @Published changes on main thread.
**How to avoid:** Always dispatch @Published property updates back to `DispatchQueue.main.async`. The computation happens on background, but the result assignment must be on main.
**Warning signs:** Purple Xcode runtime warnings in debug console.

### Pitfall 5: Measuring on Simulator vs Device
**What goes wrong:** Latency measurements on Simulator show different characteristics than physical device.
**Why it happens:** Simulator has no Taptic Engine, different GPU characteristics, and different scheduling behavior. Touch events are routed through the macOS event system.
**How to avoid:** ALL latency measurements must be on a physical iPhone. The CONTEXT.md explicitly calls for physical device measurement via Instruments (Product > Profile).
**Warning signs:** "Optimizations" that improve Simulator metrics but not device performance.

### Pitfall 6: Key Dimension Changes Breaking Layout
**What goes wrong:** Changing spacing/margins causes keys to overflow or underflow the row width, or bottom row keys misalign with top row keys.
**Why it happens:** `unitKeyWidth` is computed from available width after subtracting margins and spacing. Changing any of these shifts all key widths.
**How to avoid:** Always test layout changes across all 3 device classes. Verify that row 3 (shifted, with shift/delete taking extra width) still aligns visually with rows 1-2.
**Warning signs:** Keys appear truncated on SE, or excessive gaps on Plus/Max.

## Open Questions

1. **Exact Apple keyboard spacing values in points**
   - What we know: Apple does not publish official specs. KeyboardKit uses 51pt row height for iOS 26. Issue #29 notes side margins should roughly equal inter-key gap.
   - What's unclear: Precise inter-key gap, inter-row gap, side margin, and corner radius values per device class.
   - Recommendation: Take screenshots of Apple keyboard on each device class in Simulator, overlay a pixel ruler, and measure in points. Use these measurements as targets. This is the approach specified in CONTEXT.md (priority 2).

2. **UIViewRepresentable + SwiftUI overlay interaction for accent popup**
   - What we know: The accent popup is a SwiftUI overlay on KeyButton. With UIViewRepresentable, the touch tracking moves to UIKit but the popup rendering stays in SwiftUI.
   - What's unclear: Whether SwiftUI overlays on a UIViewRepresentable will position correctly, and whether touchesMoved in the UIView can drive SwiftUI overlay state updates smoothly.
   - Recommendation: Build a minimal prototype first. If overlay positioning breaks, fall back to having the accent popup triggered from UIKit state passed to SwiftUI via a Coordinator.

3. **Whether .drawingGroup() helps or hurts keyboard rendering**
   - What we know: drawingGroup() flattens view hierarchy to Metal texture. Good for complex static graphics, bad for independently-animating subviews.
   - What's unclear: Whether the keyboard's per-key highlight updates would benefit from or be hindered by off-screen rendering.
   - Recommendation: Do NOT use drawingGroup() initially. Only consider after profiling shows SwiftUI compositing as the bottleneck (which is unlikely given the simple key backgrounds).
