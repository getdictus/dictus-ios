  **Accent key replacement fix (KBD-02):**

  The core bug: when the user types "e" then taps the accent key showing "e", the accent is INSERTED as a new character ("ee") instead of REPLACING the previous "e" with "e".

  1. In `AccentedCharacters.swift`, add a new method:
     ```swift
     /// Returns true if the adaptive key should replace the previous character
     /// (i.e., when the key is showing an accent, not apostrophe).
     public static func shouldReplace(afterTyping lastChar: String?) -> Bool {
         guard let lastChar = lastChar?.lowercased() else { return false }
         return defaultAccents[lastChar] != nil
     }
     ```

  2. In `KeyboardView.swift`, modify the `onAccentAdaptive` callback to delete the previous character before inserting the accent:
     ```swift
     onAccentAdaptive: { char in
         HapticFeedback.keyTapped()
         // If the accent key is replacing a vowel (not inserting apostrophe),
         // delete the previous vowel first, then insert the accented version.
         if AccentedCharacters.shouldReplace(afterTyping: lastTypedChar) {
             controller.textDocumentProxy.deleteBackward()
         }
         insertCharacter(char)
     }
     ```

  3. Also fix long-press accent selection in `AdaptiveAccentKey`: when the user selects an accent from the long-press popup, it should also replace the previous vowel. Update the `.onEnded` handler in AdaptiveAccentKey to signal replacement. Change `onTap` signature to `onTap: (String, Bool) -> Void` where Bool = shouldReplace. Or simpler: keep `onTap: (String) -> Void` and let KeyboardView handle the replacement logic since it already knows `lastTypedChar`.

  Actually, the simplest approach: keep the callback as `onTap: (String) -> Void` -- the replacement logic lives entirely in KeyboardView's `onAccentAdaptive` handler. Both normal tap and long-press selection call `onTap(char)`, and KeyboardView always checks `shouldReplace(afterTyping: lastTypedChar)` before inserting. This works because `lastTypedChar` is still set to the vowel at that point (it hasn't been cleared).

  **SF Symbol icons (KBD-04 partial):**

  4. In `EmojiKey`, replace `Text("\u{1F60A}")` with `Image(systemName: "face.smiling")` using `.font(.system(size: 18, weight: .medium))`. This matches Apple's native emoji button style.

  5. In `DeleteKey`, change the SF Symbol from `"delete.left.fill"` to `"delete.backward"` (Apple's standard delete icon).

  6. In `ReturnKey`, replace `Text("retour")` with `Image(systemName: "return.left")` using `.font(.system(size: 16, weight: .medium))`. Apple uses an icon, not a text label.

  **Important:** Do NOT change the functional behavior of emoji key (it still calls advanceToNextInputMode via onTap). The user reported it goes to "Apple keyboard instead of emoji" -- this is expected behavior documented in 07-RESEARCH.md: advanceToNextInputMode() cycles to the NEXT keyboard, which may not be emoji. This is how all third-party keyboards work. No code change needed for this.
