  <name>Task 1: Improve trackpad vertical cursor movement</name>
  <files>DictusKeyboard/Views/SpecialKeyButton.swift</files>
  <action>
The current SpaceKey vertical movement uses 1-char-per-15pt, which scrolls through text linearly. Apple's trackpad moves the cursor between visual lines — a vertical drag up/down jumps roughly one line at a time.

Since `UITextDocumentProxy.adjustTextPosition(byCharacterOffset:)` is the only API available (no line-based API exists), estimate characters per line and multiply vertical offsets accordingly.

In SpaceKey, modify `handleTrackpadDrag()`:

1. Add a property to estimate characters per visible line. A reasonable estimate for iPhone is ~38-42 characters per line in most text fields (based on typical system font at standard sizes). Use 40 as default:
   ```swift
   /// Estimated characters per visible line in the host text field.
   /// Since UITextDocumentProxy has no line-width API, this is a heuristic.
   /// 40 chars is typical for iPhone body text in Messages/Notes.
   private let estimatedCharsPerLine: Int = 40
   ```

2. Change vertical movement to jump by `estimatedCharsPerLine` characters per "line unit" of vertical drag. Use ~40pt per line (roughly one text line height on screen):
   ```swift
   private let pointsPerVerticalLine: CGFloat = 40.0
   ```

3. Replace the vertical accumulation logic:
   ```swift
   // Vertical: estimate line jumps
   accumulatedOffsetY += deltaY
   let verticalLines = Int(accumulatedOffsetY / pointsPerVerticalLine)
   if verticalLines != 0 {
       let charOffset = verticalLines * estimatedCharsPerLine
       onCursorMove(charOffset)
       accumulatedOffsetY -= CGFloat(verticalLines) * pointsPerVerticalLine
   }
   ```

4. Remove the `acceleratedOffset` call for vertical movement (acceleration makes less sense for line jumps). Keep it for horizontal movement only.

This approach is not pixel-perfect but will feel much closer to Apple's behavior: a small vertical drag moves the cursor to approximately the same column on the next/previous line.

<task type="auto">
  <name>Task 2: Differentiate key sounds into 3 categories</name>
  <files>DictusKeyboard/Views/KeyboardView.swift, DictusKeyboard/Views/SpecialKeyButton.swift</files>
  <action>
Apple's keyboard has 3 distinct sounds: (a) letter keys — standard click, (b) delete key — slightly different tone, (c) modifier/special keys (space, return, shift) — another tone.

`UIDevice.current.playInputClick()` produces only one sound. To differentiate, use `AudioServicesPlaySystemSound()` with different system sound IDs. Import AudioToolbox at the top of both files.

**System sound IDs for keyboard clicks** (these are the standard iOS keyboard sounds):
- Letters: `1104` (standard keyboard click — same as playInputClick)
- Delete: `1155` (delete key sound)
- Modifier keys: `1156` (modifier key sound)

Add a helper enum in KeyboardView.swift (or at the top of the file):
```swift
import AudioToolbox

/// System sound IDs matching Apple's 3-category keyboard click sounds.
enum KeySound {
    static let letter: SystemSoundID = 1104
    static let delete: SystemSoundID = 1155
    static let modifier: SystemSoundID = 1156
}
```

Then replace `UIDevice.current.playInputClick()` calls:

In **KeyboardView.swift**:
- `insertCharacter()` (letter key callback): `AudioServicesPlaySystemSound(KeySound.letter)` — keep this for letter keys
- `onReturn` callback: Replace `playInputClick()` with `AudioServicesPlaySystemSound(KeySound.modifier)`
- `onGlobe` callback: `AudioServicesPlaySystemSound(KeySound.modifier)`
- `onLayerSwitch` callback: `AudioServicesPlaySystemSound(KeySound.modifier)`
- `onSymbolToggle` callback: `AudioServicesPlaySystemSound(KeySound.modifier)`
- `onAccentAdaptive` callback: `AudioServicesPlaySystemSound(KeySound.letter)` (accent is a letter-like key)

In **SpecialKeyButton.swift**:
- `ShiftKey` Button action: Replace `UIDevice.current.playInputClick()` with `AudioServicesPlaySystemSound(KeySound.modifier)`
- `DeleteKey` initial delete in .onChanged: Replace `playInputClick()` with nothing (sound already handled) OR replace with `AudioServicesPlaySystemSound(KeySound.delete)`. Since DeleteKey does NOT currently call playInputClick (only HapticFeedback.keyTapped), add `AudioServicesPlaySystemSound(KeySound.delete)` at the initial delete and in the repeat loop.

Note: `AudioServicesPlaySystemSound()` does NOT require UIInputViewAudioFeedback conformance or Full Access. It plays system sounds regardless. However, guard these calls with `hasFullAccess` check where already present (letter keys in KeyboardView) to respect user's sound preference. For special keys in SpecialKeyButton.swift, pass `hasFullAccess` as a parameter or add the sound call only when the feature is available.

Actually, simpler approach: since `playInputClick()` already requires UIInputViewAudioFeedback (which is set up via InputView.swift), and the keyboard already guards sound calls with `hasFullAccess`, keep the existing guard pattern. Replace `UIDevice.current.playInputClick()` with the appropriate `AudioServicesPlaySystemSound(KeySound.xxx)` call in each location, keeping the existing `if hasFullAccess` guards.

For SpecialKeyButton views (ShiftKey, DeleteKey) that don't receive `hasFullAccess`, add an `import AudioToolbox` and call `AudioServicesPlaySystemSound()` unconditionally — iOS handles the mute switch for system sounds. Actually system sounds from AudioToolbox respect the ringer switch, which is the correct behavior.
