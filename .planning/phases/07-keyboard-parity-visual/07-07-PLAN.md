  <name>Task 1: Smooth trackpad acceleration and delete word-level acceleration</name>
  <files>DictusKeyboard/Views/SpecialKeyButton.swift</files>
  <action>
  **Trackpad smoothing (KBD-01 fix):**

  The "locked to lines" feel comes from the 40-char vertical jumps at 20pt sensitivity. Apple's trackpad has smooth, proportional movement in both directions with acceleration.

  In `SpaceKey`, improve the trackpad feel:

  1. Remove the separate vertical line-jump logic. Instead, use a unified approach where vertical drag translates to proportional cursor offset (not discrete 40-char jumps).

  2. Add acceleration: as the drag velocity increases, increase the cursor movement multiplier. Calculate velocity from delta between current and previous drag positions.

  3. Replace the fixed `pointsPerLine: CGFloat = 20.0` with a smoother vertical sensitivity:
     - Reduce vertical jumps from 40 chars to a smaller, proportional value
     - Use `pointsPerVerticalChar: CGFloat = 15.0` and move cursor by individual characters vertically (e.g., 1 char offset per 15pt of vertical drag), not 40-char line jumps
     - This creates smooth vertical scrolling through text, not jarring line jumps

  4. Add a simple velocity-based acceleration curve:
     ```swift
     private func acceleratedOffset(_ rawDelta: CGFloat, sensitivity: CGFloat) -> Int {
         let baseChars = rawDelta / sensitivity
         // Apply acceleration: faster drag = more characters per point
         let velocity = abs(rawDelta)
         let multiplier: CGFloat = velocity > 20 ? 2.0 : (velocity > 10 ? 1.5 : 1.0)
         return Int(baseChars * multiplier)
     }
     ```

  5. Keep the existing haptic pattern unchanged (tap on press, medium on activate, none during drag).

  **Delete acceleration (from KEYBOARD_BRIEFING.md):**

  In `DeleteKey`, add word-level deletion after sustained hold:

  1. Add a `@State private var deleteCount: Int = 0` counter.

  2. After the initial 400ms delay, in the repeat loop:
     - First 10 deletions: `deleteBackward()` one character at a time (current behavior)
     - After 10 deletions: call a word-delete approach. Since `textDocumentProxy` does not have `deleteWordBackward()`, implement it manually:
       - Read `documentContextBeforeInput`, find the last word boundary (space or start of string)
       - Call `deleteBackward()` for each character back to that boundary
     - Increase repeat interval slightly for word mode (120ms instead of 100ms) to give visual feedback

  3. Reset `deleteCount = 0` in `.onEnded`.

  4. Add haptic on each deletion step: `HapticFeedback.keyTapped()` inside the repeat loop (matches Apple behavior where delete has subtle haptic).

  <action>
  Implement automatic shift activation after sentence-ending punctuation, per KEYBOARD_BRIEFING.md.

  **Autocap logic in KeyboardView:**

  1. Add a private method to KeyboardView:
     ```swift
     private func checkAutocapitalize() {
         // Respect the host app's autocapitalizationType
         // (some fields like email disable autocap)
         let before = controller.textDocumentProxy.documentContextBeforeInput ?? ""

         let shouldCap: Bool
         if before.isEmpty {
             shouldCap = true  // Empty field = capitalize
         } else if before.hasSuffix(". ") || before.hasSuffix("! ") || before.hasSuffix("? ") {
             shouldCap = true  // After sentence-ending punctuation + space
         } else if before.hasSuffix("\n") {
             shouldCap = true  // After newline
         } else {
             shouldCap = false
         }

         // Only auto-activate shift, don't interfere with caps lock
         if shouldCap && shiftState == .off {
             shiftState = .shifted
         } else if !shouldCap && shiftState == .shifted {
             // Don't auto-deactivate if user manually shifted
             // Actually, auto-unshift is already handled by insertCharacter()
             // So only auto-SHIFT, never auto-unshift here
         }
     }
     ```

  2. Call `checkAutocapitalize()` after:
     - `insertCharacter()` completes (after a space is inserted following punctuation)
     - `onSpace` callback (space after `. ` could mean autocap)
     - `onReturn` callback (newline = capitalize)
     - `onDelete` callback (deleting may expose a sentence ending)
     - View `.onAppear` (initial field state)

  3. In KeyboardView's `.onAppear`, add `checkAutocapitalize()` to set initial shift state when keyboard appears.

  **KeyboardViewController integration:**

  4. In `textDidChange(_:)`, post a notification or use a shared state mechanism to trigger autocap recheck when the text field content changes externally (e.g., user pastes text, or cursor moves). The simplest approach: since KeyboardView already has a reference to `controller`, and `textDidChange` fires after external changes, add a `@Published` property to a shared observable or use a simple notification:

     - Add a `NotificationCenter` post in `textDidChange`:
       ```swift
       override func textDidChange(_ textInput: UITextInput?) {
           super.textDidChange(textInput)
           NotificationCenter.default.post(name: .dictusTextDidChange, object: nil)
       }
       ```
     - Add the notification name extension:
       ```swift
       extension Notification.Name {
           static let dictusTextDidChange = Notification.Name("dictusTextDidChange")
       }
       ```
     - In KeyboardView, listen for this notification with `.onReceive(NotificationCenter.default.publisher(for: .dictusTextDidChange))` and call `checkAutocapitalize()`.

  5. Respect `autocapitalizationType`: check `controller.textDocumentProxy.autocapitalizationType`. If it's `.none`, skip autocap entirely. If `.words`, capitalize after every space. If `.sentences` (default) or `.allCharacters`, apply the appropriate logic. For simplicity, implement `.sentences` (the most common) and `.none` -- those cover 95% of use cases.
