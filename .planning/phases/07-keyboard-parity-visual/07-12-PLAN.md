<task type="auto">
  <name>Task 1: Fix waveform breaking after cancel</name>
  <files>DictusApp/DictationCoordinator.swift</files>
  <action>
**Root cause:** `cancelDictation()` calls `audioRecorder.stopRecording()` which sets `isEngineRunning = false` and stops the WhisperKit audio engine entirely. The next recording attempt from the keyboard (via Darwin notification from background) hits the cold start path in `AudioRecorder.startRecording()`, which calls `startRecordingLive()` — but this requires foreground context. Since the app may be in background, the engine fails silently, no audio data flows, and the waveform shows nothing.

**Fix:** In `cancelDictation()`, replace `audioRecorder.stopRecording()` with `audioRecorder.collectSamples()`. The `collectSamples()` method:
- Returns the audio samples (which we discard with `_ =`)
- Sets `isRecording = false`
- Resets `bufferEnergy` and `bufferSeconds`
- Does NOT stop the audio engine (`isEngineRunning` stays `true`)
- Keeps the app alive in background via UIBackgroundModes:audio

This matches the design philosophy already documented in AudioRecorder: "the engine is kept running between recordings (Wispr Flow pattern)". Cancel should follow the same pattern as stop — discard the audio data but keep the engine warm.

Change in `cancelDictation()`:
```swift
func cancelDictation() {
    dictationTask?.cancel()
    dictationTask = nil

    // Discard samples but keep engine alive for next recording.
    // WHY collectSamples() not stopRecording(): stopRecording() kills the
    // audio engine, which means the next recording from background fails
    // (cold start requires foreground). collectSamples() discards audio
    // without stopping the engine — same pattern as normal stop flow.
    _ = audioRecorder.collectSamples()

    // Reset all state
    bufferEnergy = []
    bufferSeconds = 0
    cleanupRecordingKeys()
    updateStatus(.idle)

    if #available(iOS 14.0, *) {
        DictusLogger.app.info("Dictation cancelled by keyboard")
    }
}
```

That is the only change needed. The keyboard side (`KeyboardState.requestCancel()`) already resets `waveformEnergy = []` correctly.

  <name>Task 2: Document emoji button as iOS limitation</name>
  <files>DictusKeyboard/Views/SpecialKeyButton.swift</files>
  <action>
The emoji button calls `advanceToNextInputMode()` which cycles to the next installed keyboard, not specifically the emoji keyboard. There is no iOS API to target the emoji keyboard directly. This is the same behavior used by Gboard, SwiftKey, and all third-party keyboards.

Since this is documented in 07-RESEARCH.md as a known limitation (Pitfall 5, Open Question 3), and the VERIFICATION.md acknowledges "May be an iOS limitation", accept this and update the EmojiKey documentation:

1. Update the doc comment on `EmojiKey` struct to clearly state:
   ```swift
   /// Emoji key — shows a face.smiling icon matching Apple's native AZERTY visual style.
   /// Tapping cycles to the next installed input mode via advanceToNextInputMode().
   ///
   /// KNOWN iOS LIMITATION: No public API exists to target the emoji keyboard
   /// specifically. advanceToNextInputMode() cycles through ALL installed keyboards
   /// in order. This matches Gboard, SwiftKey, and other third-party keyboard behavior.
   /// The emoji icon is used because Apple's native AZERTY shows this icon, and in most
   /// user configurations, the next input mode IS the emoji keyboard.
   ```

2. No functional code changes — the behavior is correct given iOS constraints.
