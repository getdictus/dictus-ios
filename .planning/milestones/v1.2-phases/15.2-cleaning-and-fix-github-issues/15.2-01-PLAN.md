  <action>
The waveform data arrives correctly via App Group (confirmed by logs), but BrandWaveform does not visually animate in the keyboard extension after process suspension. Root cause: after iOS suspends/resumes the keyboard extension process, TimelineView's internal CADisplayLink may not restart, and the onChange(of: energyLevels) fallback only calls updateDisplayLevels() once per data change (~5Hz), producing micro-changes that SwiftUI coalesces away.

**In BrandWaveform.swift:**

1. Add a `@State private var renderTick: Int = 0` property alongside `displayLevels`. This is a monotonically increasing counter that forces SwiftUI to re-evaluate the Canvas even when displayLevels changes are too small to trigger a diff.

2. In `updateDisplayLevels()`, after updating the `displayLevels` array, increment `renderTick`:
```swift
displayLevels = updated
renderTick += 1
```

3. In `waveformContent(processingPhase:)`, reference `renderTick` inside the Canvas closure to create a dependency that forces redraw:
```swift
Canvas { context, size in
    let _ = renderTick  // Force redraw on each tick
    let totalSpacing = barSpacing * CGFloat(barCount - 1)
    // ... rest unchanged
```

4. In the `onChange(of: energyLevels)` handler (the fallback path, line ~90), ensure it fires even when array equality might match by adding a small perturbation approach. Change:
```swift
.onChange(of: energyLevels) { _ in
    if !isProcessing {
        updateDisplayLevels()
    }
}
```
to:
```swift
.onChange(of: energyLevels) { newLevels in
    if !isProcessing {
        // Force display update even if SwiftUI considers arrays equal
        // (can happen when micro-changes are below Float equality threshold)
        updateDisplayLevels()
    }
}
```
(The renderTick increment inside updateDisplayLevels handles the forced redraw, so the onChange handler itself stays clean.)

**In KeyboardState.swift:**

5. In the method that reads waveform data from App Group (find `refreshFromDefaults()` or the Darwin notification handler that reads `SharedKeys.waveformEnergy`), add an explicit `objectWillChange.send()` call AFTER updating `waveformEnergy`:
```swift
// After setting waveformEnergy from App Group data:
self.waveformEnergy = energy
self.objectWillChange.send()  // Force SwiftUI to re-evaluate even if array equality passes
```

This ensures the KeyboardState -> BrandWaveform data flow triggers a view update even when iOS's extension process resumes with a potentially stale SwiftUI observation chain.

Add a PersistentLog entry in the waveform data read path for diagnostic verification:
```swift
PersistentLog.log("Waveform data received in keyboard: \(energy.count) levels, renderTick will increment")
```
