#### 1. Haptic and audio timing on physical device

**Test:** Tap letter keys, delete, space, and return on a physical iPhone running Dictus

**Test:** Long-press the "e" key for 400ms, then slide finger to select an accent
**Expected:** Accent popup appears above the key. Sliding left/right highlights the correct accent. Releasing inserts the accent at cursor position.
**Why human:** Touch interaction flow cannot be tested programmatically.

### Gaps Summary

**One gap blocks Success Criterion 1:** The UIViewRepresentable approach for letter key touch handling was created (Plans 01-02) but then reverted (commit `a296c13`) because edge-position keys did not receive touches due to SwiftUI's frame-boundary clipping. The current solution — `DragGesture(minimumDistance: 0)` with `contentShape(Rectangle())` — is documented in a code comment as the correct trade-off.

**Impact of gap:** The stated goal mentions "UIKit touch handling" as a mechanism. The actual latency improvement was achieved through a different path: audio/haptic on touchDown, suggestions on background queue, static KeyMetrics. The ~5-10ms latency difference from UIViewRepresentable vs DragGesture was deemed "imperceptible" in the revert commit message. The phase goal (reduce input latency, match Apple keyboard feel) was substantially achieved through the other mechanisms.

**Action required:** Either (a) update the ROADMAP Success Criterion 1 to reflect the architectural decision (DragGesture with contentShape is the correct approach), or (b) find a working UIKit touch solution that handles edge-key dead zones. The current codebase is internally consistent — the code comment and the implementation agree. Only the ROADMAP criterion is stale.
