**Part A: Swipe-to-delete**

Implement swipe-to-delete for model cards in ModelManagerView per user decision (like iOS Mail).

**Approach decision:**
The research notes that `.swipeActions` requires `List` context, and ModelManagerView currently uses `ScrollView > VStack > ForEach`. Two options:

**Option A (preferred -- try first):** Convert to `List` with transparent styling:
```swift
List {
    ForEach(models) { model in
        ModelCardView(...)
            .swipeActions(edge: .trailing, allowsFullSwipe: false) {
                // Only show delete for downloaded, non-active models
                if canDelete(model) {
                    Button("Supprimer", role: .destructive) {
                        modelToDelete = model
                        showDeleteAlert = true
                    }
                }
            }
            .listRowBackground(Color.clear)
            .listRowSeparator(.hidden)
            .listRowInsets(EdgeInsets(top: 6, leading: 16, bottom: 6, trailing: 16))
    }
}
.listStyle(.plain)
.scrollContentBackground(.hidden)
```

**Option B (fallback if List breaks glass aesthetic):** Use `.contextMenu` on each card instead of swipe:
```swift
ModelCardView(...)
    .contextMenu {
        if canDelete(model) {
            Button("Supprimer", role: .destructive) { ... }
        }
    }
```

**Delete guard logic:**
- Model must be downloaded
- Model must NOT be the active model
- There must be more than 1 downloaded model (cannot delete last one)
- Show confirmation alert before deletion: "Supprimer [model name] ? Le mod\u00e8le sera supprim\u00e9 de votre appareil."

**State management:**
- Add `@State private var modelToDelete: ModelInfo? = nil`
- Add `@State private var showDeleteAlert = false`
- Use `.alert()` modifier for delete confirmation
- On confirm, call `modelManager.deleteModel(model.identifier)` (verify this method exists, if not use the appropriate cleanup method)

Also remove any existing visible delete/trash buttons from ModelManagerView if they exist separately from ModelCardView.

Note: This fix is placed here because Plan 02 already owns ModelManagerView.swift. Plan 04 handles the other half of bug #25 (ensuring ModelDownloadPage persists state correctly).
