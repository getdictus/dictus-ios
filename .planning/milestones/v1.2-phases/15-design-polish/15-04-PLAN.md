**Bug #25: Onboarding model not recognized on Models page:**

Root cause from research: ModelDownloadPage creates its own `@StateObject private var modelManager = ModelManager()`. This is a SEPARATE instance from ModelManagerView's instance. Both write to App Group via `persistState()`, but ModelManagerView only reads state in `init()`.

Fix approach:
1. Read ModelManager.swift to find the `loadState()` method (or equivalent)
2. Ensure `loadState()` is a public method (if not, make it public)
3. Verify that ModelDownloadPage correctly calls `modelManager.persistState()` after download completes. If it doesn't, add the call
4. Note: The other half of this fix (ModelManagerView calling loadState() on appear) is handled in Plan 02 Task 2 which owns ModelManagerView.swift

**Bug #26: Crash on return from keyboard settings:**

Root cause from research: Intermittent crash (~1 in 2-3) when returning from iOS Settings after enabling keyboard + Full Access during onboarding. Race condition in scenePhase / didBecomeActive during keyboard status check.

Fix approach:
1. Read KeyboardSetupPage.swift to find `checkKeyboardInstalled()` and `onChange(of: scenePhase)` handler
2. Add PersistentLog around the keyboard detection code for diagnosis
3. Wrap `UITextInputMode.activeInputModes` access in a guard
4. Debounce the keyboard detection by adding a small delay:
   ```swift
   .onChange(of: scenePhase) { _, newPhase in
       if newPhase == .active {
           Task {
               try? await Task.sleep(for: .milliseconds(500))
               await MainActor.run {
                   checkKeyboardInstalled()
               }
           }
       }
   }
   ```
5. Add a `@State private var isCheckingKeyboard = false` guard to prevent concurrent checks
6. Log the scenePhase transitions and keyboard detection results for future debugging
